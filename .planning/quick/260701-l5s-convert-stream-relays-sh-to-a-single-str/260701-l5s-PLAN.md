## Goal

Replace the 174-process `strfry stream` fan-out in `stream-relays.sh` (one
`docker exec strfry /app/strfry stream <relay>` per relay) with a **single**
`strfry router` process, to eliminate the multi-process LMDB reader-slot
contention that surfaced as `MDB_BAD_RSLOT` crashes on the arm64 host after the
bloom writePolicy plugin was fixed (see the strfry-host debug thread).

Hard requirement: downloaded ("down") events MUST stay gated by the whitelist
bloom plugin. Source review (see RESEARCH.md) confirmed the global `writePolicy`
does NOT run on router-down events — each stream needs `pluginDown = "/app/plugins/bloom"`,
or ingestion is silently ungated (a whitelist regression).

## Intended implementation (once the blocker below is resolved)

`cmd_start` in `stream-relays.sh` would, instead of N tmux windows each running
`strfry stream`:
1. Generate a streams-only router config from the resolved relay list:
   ```
   streams { ingest { dir = "down"  pluginDown = "/app/plugins/bloom"  urls = [ ... ] } }
   ```
2. Launch ONE `strfry router <config>` process in a single tmux window.
3. Keep the existing relay-loading (web-of-trust.yaml + fallback) and stop/attach subcommands.

## BLOCKER (why this is not implemented)

`strfry router` on the deployed **dockurr/strfry 1.1.0** (pinned commit
`f31a1b9`) will not activate any streams, despite a source-verified config +
invocation. Verified on the host (~9 variants):

- `strfry router /tmp/router.conf` with a valid streams-only file (top-level
  `streams{}`, correct per source) — strfry receives the positional arg
  (confirmed in the `arguments:` log line), the file exists and is valid, but
  the router **never emits `Loading router config file:`**, never opens
  connections, never spawns the `pluginDown` bloom process, pulls 0 events, logs
  no error, and idles. Only the 7 startup lines appear.
- `strfry --config=<file-with-streams-appended> router` — golpe's `loadConfig()`
  silently drops the unknown `streams` top-level key ("CONFIG: successfully
  installed"), so streams never register either way.
- `connections{}` (the first researched key) behaved identically.

Net: the positional router config is not being consumed by this binary. Root
cause is build-specific and unresolved by black-box testing.

## Recommended next steps

1. Read dockur's exact router wiring (`src/apps/mesh/cmd_router.cpp` + the docopt
   USAGE string) to see whether `router` declares a positional `<routerConfigFile>`
   in this build, or expects the config another way.
2. Try dockur's documented entrypoint env var `ROUTER=/path/to/router.conf`
   (its entrypoint runs `./strfry router "$ROUTER"`) as a separate container/service,
   rather than `docker exec` into the relay container.
3. Raise strfry stderr verbosity to see post-config router behavior.
4. If router cannot be made to work on this pinned image, consider the durable
   alternative: move the LMDB to a Docker named volume (Linux ext4) so the shared
   reader table tolerates the multi-process `stream` fan-out (addresses the
   MDB_BAD_RSLOT root cause directly).

## Not done

`stream-relays.sh` is UNCHANGED — no unverified ingestion rewrite was shipped.
