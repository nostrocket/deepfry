<objective>
Close gap Cluster 3 (CR-06, CR-07 + WR-04, IN-02): tag filter semantics are wrong. The router
hex-decodes ANY even-length hex tag value (CR-07) — a literal topic tag like "beef" gets decoded to
0xBE 0xEF and the scan lands at the wrong index position, silently missing the real entries. The
engine evaluates multiple TagFilters with OR semantics (CR-06) — `{#e:[X], #p:[Y]}` returns events
with X OR Y instead of NIP-01's required AND across distinct fields. The tag name is also silently
truncated to its first byte (WR-04).

Fix: restrict the Event__tag hex-decode to exactly 64-char lowercase hex (strfry's 32-byte-id
rule); validate single-character tag names (skip with warn otherwise); change the engine tag
residual to AND across distinct TagFilter fields (values within one field stay ORed); build scan
keys from a single TagFilter's values (the scan driver), with the other filters enforced as
residual.

Purpose: Restore QRY-02 — tag filters return events matching the given tag name and values, with
correct NIP-01 AND semantics and strfry-byte-identical key construction.
Output: 64-char-lowercase-hex-only tag decode, tag-name validation, AND-semantics tag residual,
shared hex helper, and regression tests for literal-hex tag values, multi-tag AND, and bad tag names.
</objective>

| tag filter values → Event__tag key bytes | Untrusted callers supply tag values; wrong hex/UTF-8 decode places the scan at the wrong index position (silently missing matches) |
| tag filter set → result set | OR-vs-AND semantics determine whether events the caller did not ask for are returned |
