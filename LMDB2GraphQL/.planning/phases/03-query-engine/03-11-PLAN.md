<design_notes>
## CR-02 fix choice — why "detect no-progress and break", NOT "thread lev_id into the scan bound"

The prompt asks to evaluate both CR-02 options against D-05/D-10/D-11 and the merge_windowed
contract, and pick the one that preserves correct DESC (created_at, lev_id) ordering without
skipping events at the boundary. The analysis is settled here so the executor does not re-litigate:

**Threading lev_id into the start_key is NOT possible** for the fat-timestamp case. Every `Event__*`
composite key is `prefix ‖ created_at(8 LE)` — `lev_id` is the DUPSORT *value*, NOT part of the key
(spec §3.4; confirmed in merge.rs `created_at_from_key` and scan.rs key-layout docs). `build_start_keys`
can only position a scan at `(prefix, created_at)`; it cannot position *within* a dup group by lev_id.
So re-running `merge_windowed` with `until = ts_L` always re-scans the dup group from its top. There is
no key byte to encode `lev_id` into. (`scan_index_one_window`'s `Bound::Excluded(resume_key)` resume is
key-granular too — excluding the last *key* skips the whole `ts_L` dup group, which would skip
un-emitted dups. The merge's internal within-window resume cannot be carried across the engine's
per-round restart because each round rebuilds start_keys from scratch.)

**Therefore the correct, minimal, order-preserving fix is the no-progress break** (the review's
explicit safety-net option), composed with the CR-01 deepest_scanned cursor:

1. Each round, after computing `last_merged`, if `last_merged == round_boundary` the round made zero
   forward progress (it re-scanned the same prefix and cursor-exclusion dropped everything). `break`
   the loop instead of advancing `round_boundary` to the identical value and spinning to MAX_ROUNDS.
2. The CR-01 cursor builder then returns a `Some(...)` cursor (from `valid.last()` if any survivor was
   collected this page, else from `deepest_scanned`) pointing at the deepest scanned `(ts, lev_id)`.
3. The caller's NEXT page restarts the merge with `until = ts_L` and the cursor-exclusion
   `ts == cur_ts && lev_id >= cur_lev` drops the already-emitted top of the `ts_L` dup group, so the
   next page reaches the lower-lev_id dups and eventually descends past `ts_L`. Cross-page progress is
   monotonic in `(created_at, lev_id)` DESC; no event is skipped and none is double-emitted (the
   `>= cur_lev` exclusion is inclusive of the boundary lev_id and excludes it).

This preserves D-10 ordering (the merge still emits `(created_at DESC, lev_id DESC)`), honors D-11
(resume by the opaque `(created_at, lev_id)` cursor), and respects D-05's per-prefix merge contract
(merge.rs is untouched). It does NOT skip events at the boundary: within-page un-emitted dups become
reachable on the following page via cursor-exclusion, exactly as the engine already pages a normal
under-full result.

**Why both fixes are one plan:** CR-01 and CR-02 both live in the round-loop and the single
`next_cursor` builder. The CR-02 no-progress break is only safe *because* the CR-01 deepest_scanned
fallback guarantees a non-None cursor when `valid` is empty. Splitting them would let a CR-02 break
fire while the cursor builder still returns None (re-introducing CR-01's stranding). They are coupled;
keep them together with clear task boundaries (fix in Task 1, regression coverage in Task 2).
</design_notes>
