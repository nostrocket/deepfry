<objective>
Add permissive wildcard CORS to the existing axum GraphQL server so a browser frontend on any origin can query POST /graphql (and GET /graphql for GraphiQL) cross-origin, with correct preflight handling and no weakening of existing protections.

Purpose: v1.1 milestone — unblock a browser-based frontend served from a different host. The corpus is public, unauthenticated, read-only Nostr data, so wildcard origin with NO credentials is the intended, safe posture.

Output: tower-http `cors` feature enabled (no new crate); a single `CorsLayer` added as the outermost layer in `build_router`; an updated rustdoc reflecting the new outermost layer; and a `tests/cors_test.rs` integration suite proving CORS headers on 200/413/503 plus a preflight.

This phase touches neither the LMDB read path nor any datastore — it only adds an HTTP response-header layer. The DeepFry data-separation rule and the read-only LMDB constraint are unaffected.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser (any origin) → HTTP socket (`bind_address`) | Untrusted cross-origin browser requests cross here. CORS relaxes ONLY the browser same-origin policy for responses; it does NOT change who can route to the socket. |
| HTTP request body → RequestBodyLimitLayer → handler | Untrusted POST body; capped at 256 KiB (413) — unchanged by this phase. |
| POST /graphql → OnceCell schema gate | Corpus reachability gate; 503 until schema populated — unchanged by this phase. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Information Disclosure | CorsLayer wildcard + credentials | accept (mitigated by design) | Never call `allow_credentials`; the API is unauthenticated with no cookies/tokens. tower-http additionally `assert!`-panics if `allow_credentials(true)` is combined with `Any`, so the unsafe combo fails closed at startup (CORS-03). Test asserts no `Access-Control-Allow-Credentials` header on any response. |
| T-06-02 | Elevation of Privilege / Info Disclosure | CORS misread as a network ACL → operator binds `0.0.0.0` | accept | CORS is NOT a network access control. The `bind_address` loopback default and the non-loopback `tracing::warn!` in main.rs remain the network controls and are NOT modified this phase (CORS-04). Rustdoc states CORS relaxes browser same-origin policy only. |
| T-06-03 | Denial of Service (functional) | OPTIONS preflight reaching a body/auth-requiring handler | mitigate | CorsLayer short-circuits OPTIONS before routing; no manual OPTIONS handler is added. Preflight never descends to the body-limit layer or the schema-gated handler (CORS-02). |
| T-06-04 | Denial of Service | Oversized cross-origin POST | mitigate | `RequestBodyLimitLayer` (256 KiB) is unchanged and stays inside CorsLayer, so the 413 still fires AND still carries CORS headers (clean rejection, no browser CORS-error masking). Test `test_cors_headers_on_413` asserts both (CORS-04). |
| T-06-05 | Tampering | npm/pip/cargo installs | mitigate | No new crate is added — only the already-vendored `tower-http` (verdict OK, RESEARCH.md Package Legitimacy Audit) gains its `cors` feature flag. No package-legitimacy human-verify checkpoint required. |

**Net delta:** No `high`-severity threats. CORS relaxes the browser same-origin policy on an already-public, unauthenticated, read-only endpoint. No credentials/cookies/auth are exposed; the body-cap (413), the 503-readiness gate, and the loopback-bind default are preserved verbatim and asserted by tests (CORS-04).
</threat_model>
