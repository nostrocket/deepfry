# Quick Task 260615-1e6: Fixture-free comparator self-check

## Problem

`main.rs` runs `run_comparator_self_check(&env, &golden)` against the **live production
strfry env**, but the self-check's Phase 1 (`src/lmdb/self_check.rs`) asserts the *entire*
levId sequence of each `Event__*` index equals the committed **11-event fixture golden
vector**. On any real DB this mismatches on the first index, the `OrderMismatch` error path
logs `actual = ?actual` (the whole million-entry index → ~243 MB to stderr), and the process
exits non-zero — it never becomes ready. The self-check is fixture-only by design (Phase 2's
adversarial seeks also use fixture-only keys: `kind=256`, generator pubkey `79be…`).

## Goal

Redesign the self-check so it validates comparator correctness against **any** live strfry
DB with **no fixture and no golden vectors at runtime**, keeps failing closed on a genuine
comparator fault, and can never dump unbounded data into a log.

## Approach (the oracle is the physical B-tree order)

`db.iter()` walks the B-tree in the physical order strfry's *real* golpe comparator
established when it built the tree. That order is the oracle — no golden sequence needed.

Per `Event__*` index, over a **bounded sample** (cap ~10k entries; keep read txns short per
the LMDB safety constraint — short per-window txns, no long-lived `RoTxn`):

- **Phase A — FFI/logic correctness.** Walk physical order via `db.iter()` collecting KEYS.
  For each adjacent pair assert `Cmp::compare(prev, cur) != Ordering::Greater` (non-decreasing;
  `MDB_DUPSORT` means adjacent equal keys are valid). A violation ⇒ fail closed. This proves
  the registered golpe FFI comparator reproduces the tree's order on **real** keys (catches
  wrong-comparator-on-index and FFI/ABI/endianness drift).
- **Phase B — heed registration / memcmp-fallback detection.** For a bounded key sample
  (prioritize keys adjacent to **witness pairs** — adjacent physical pairs where
  `memcmp(prev,cur) == Greater` while physical order is `prev < cur`), do
  `db.range(K..).next()` and assert the first returned key `== K`. A correctly-registered
  comparator always round-trips an existing key; a silent memcmp fallback on a golpe-ordered
  tree misses for divergent keys. If an index's sample yields **zero** witness pairs, emit a
  `tracing::warn!` that registration could not be *adversarially* verified on this dataset
  (the exact-seek round-trip still holds).

`Event__created_at` uses `heed::IntegerComparator` (not a golpe FFI comparator) — apply both
phases with that comparator.

  - Remove the embedded golden-vector `include_str!` block, the `GoldenVectorJson`/`GoldenVectors`
