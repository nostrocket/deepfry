<objective>
Close CR-01: the reverse-scan first-window / bounded-scan `Bound::Included(start_key)` on an
EXISTING boundary key silently drops the higher duplicates of that key. heed 0.22.1's
`rev_range(Bound::Included(K))` with `move_through_duplicate_values()` positions at the SMALLEST
dup of K and steps to the previous key — higher dups of K are never yielded (PROVEN by
tests/dupsort_resume_test.rs::test_old_code_reverse_drops_levid_nonvacuity).

This plan fixes the two Reverse arms in src/lmdb/scan.rs that still use `Bound::Included` on the
caller-supplied finite start bound: `collect_window` first-batch arm (lines ~461-469) and
`collect_bounded` Reverse arm (lines ~370-374). The fix builds the reverse upper bound from
`ts + 1` (saturating at u64::MAX) and uses `Bound::Excluded(key_with_ts_plus_one)` so SET_RANGE
positions ABOVE the boundary key, walks back, and lands on the LARGEST dup of the real `ts` —
the full dup group is then walked descending.

Purpose: NIP-01 `until` is inclusive and cursor page boundaries land on real timestamps; this
is a data-loss path on any production database with dup groups of size >= 2 (and the
Event__created_at default feed where the dup group is every event in a second).
Output: corrected scan.rs Reverse bound construction + regression tests against the fixture and
synthetic 3-dup indexes.
</objective>
