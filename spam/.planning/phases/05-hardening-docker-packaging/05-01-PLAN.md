<objective>
Add the in-process operational surface: `/health` and `/ready` HTTP probes (OPS-01) and version-drift surfacing through the `stats` GraphQL query (OPS-04). `/health` always returns 200 when the process is alive; `/ready` returns 200 only after the LMDB env is open and `run_comparator_self_check` has passed, 503 otherwise. The `stats` query additionally returns the pinned strfry image reference alongside the already-detected on-disk `dbVersion`, so operators can spot drift if the parent `dockurr/strfry` image moves.

Purpose: Orchestrators and operators need an independent liveness/readiness signal, and a queryable record of which strfry version the adapter is pinned against. These are the runtime prerequisites for the Docker packaging in Plan 05-02 (the docker-compose healthcheck hits `/health`).
Output: Modified `server.rs`, `main.rs`, and the three GraphQL files; a new integration test file proving the probe semantics.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client тЖТ /health, /ready | Unauthenticated HTTP probes cross here; take no input |
| client тЖТ /graphql stats | Unauthenticated read-only query reads operational metadata (pinned version, dbVersion) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Information disclosure | `/ready` endpoint leaking internal startup state | accept | `/ready` exposes only a boolean (ready/not-ready) as an HTTP status code; no internal detail, paths, or error text in the body. ASVS L1 V4 partial тАФ endpoint is an internal probe by design. |
| T-05-02 | Information disclosure | `stats.pinnedStrfryVersion` reveals deployed strfry image digest | accept | The pinned image reference is internal operational data, not a secret. The existing CR-01 non-loopback-bind warning in main.rs already governs whether the endpoint is publicly reachable; loopback default keeps it private. |
| T-05-03 | Denial of service | unauthenticated `/health` probe flood | accept | `health_handler` does zero work (returns a constant StatusCode, no LMDB access, no allocation) тАФ cheapest possible handler; the existing RequestBodyLimitLayer still bounds POST bodies. Low-value target per security domain. |
| T-05-04 | Tampering | readiness flag set before gates pass (false-ready) | mitigate | `ready` initialized `false`; `store(true, Ordering::Release)` placed strictly AFTER `run_comparator_self_check` in main.rs; acceptance criterion asserts source ordering. Prevents serving "ready" while LMDB/comparator state is unverified. |
| T-05-SC | Tampering | npm/pip/cargo installs | mitigate | No new Cargo dependencies in this plan (RESEARCH "No new Cargo dependencies required"); uses std `AtomicBool`, existing `axum`/`tokio`/`async-graphql`. No package legitimacy gate needed. |
</threat_model>
