<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| host filesystem → container | strfry-db bind-mount crosses here; must be read-only |
| container network → host | published port crosses here; loopback-only |
| GitHub Actions runner → registry | pulls the pinned strfry image by immutable digest |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-05 | Tampering | strfry-db mount becoming writable corrupts strfry's env | mitigate | Mount appended with `:ro` (read-only); acceptance criterion greps for `:/app/strfry-db:ro`. The adapter never opens a write txn (MDB_RDONLY), and `:ro` enforces it at the kernel mount level even if a bug tried. |
| T-05-06 | Information disclosure | GraphQL/GraphiQL/introspection reachable beyond the host | mitigate | docker-compose publishes `127.0.0.1:8080:8080` (loopback only, not `0.0.0.0:8080:8080`); the in-container `bind_address: 0.0.0.0:8080` only binds inside the container namespace — the host publish rule is the actual exposure control. The CR-01 non-loopback warning in main.rs covers misconfiguration. ASVS L1 V4. |
| T-05-07 | Tampering | strfry image drift (mutable tag) yields wrong comparators/fixture | mitigate | Both the Dockerfile-adjacent fixture pipeline and CI pull strfry by immutable `@sha256:` digest, not a mutable tag; the digest matches config.rs `pinned_strfry_version` and Dockerfile.strfry. CI `cargo test` fails closed if the regenerated fixture diverges from the committed golden vectors. |
| T-05-08 | Availability | `/ready` used as docker healthcheck causes restart loop | mitigate | Healthcheck targets `/health` (constant 200), not `/ready`; `start_period: 10s` gives the LMDB self-check time before health counts. RESEARCH Pitfall 5. |
| T-05-09 | Tampering | supply-chain: CI installs / image pulls | mitigate | No new Cargo deps. Rust toolchain via official `sh.rustup.rs` over TLS; strfry pulled by pinned digest; Alpine apk packages are standard distro packages. No `[ASSUMED]`/`[SUS]` package installs requiring a legitimacy checkpoint. |
| T-05-SC | Tampering | npm/pip/cargo installs | mitigate | This plan adds no Cargo dependencies (RESEARCH "No new Cargo dependencies required"). apt/apk install only OS build deps (liblmdb-dev, g++, pkg-config, musl-dev) from official distro repos. |
</threat_model>
