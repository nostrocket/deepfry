<objective>
Close the single BLOCKER gap from 05-VERIFICATION.md: the OPS-01 readiness window is unreachable in production. The `/ready` handler logic in `server.rs` is already correct, but `main.rs` flips the readiness flag to `true` (line 85) BEFORE the TcpListener binds (line 101) and BEFORE `axum::serve` starts (line 122). The socket does not exist until after the flag is already true, so no external orchestrator can ever observe a 503 from `/ready`. The 503 branch is dead code in production.

This plan restructures the `main.rs` startup ordering so a probe-capable HTTP surface (`/health` + `/ready`, with `ready=false`) is bound and serving on `cfg.bind_address` BEFORE the gate chain (LMDB env open → Meta gates → comparator self-check) runs. The readiness flag flips to `true` only after `run_comparator_self_check` returns `Ok`. The server then transitions to serving the full GraphQL router. The 503→200 transition becomes observable to a real client polling the live socket during startup, and fail-closed behaviour (V7) is preserved.

Purpose: The phase goal requires that "health/readiness gates are live." OPS-02/03/04 are already SATISFIED — this plan touches ONLY the startup ordering needed to make the /ready window reachable.
Output: A new `build_probe_router` in `server.rs`, a restructured `main.rs` startup sequence, and a regression test (`ready_window_test.rs`) that exercises the 503→200 transition through a router-served surface — the blind spot the existing handler-only unit tests missed.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → /health, /ready (gate window) | Unauthenticated HTTP probes cross here DURING startup, before gates pass; take no input |
| client → /health, /ready, /graphql (post-gate) | Unchanged from 05-01; the full router serves only after gates pass and ready=true |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-05-SC | Information disclosure | A degraded/partial server serving data routes while `ready=false` (during the gate window) | mitigate | The startup-window surface is `build_probe_router`, which mounts ONLY `/health` and `/ready` — NO `/graphql` route and NO GraphQL state. The full `build_router` (with the corpus-reading GraphQL service) is constructed and served ONLY after `run_comparator_self_check` returns `Ok` and the probe server has been gracefully shut down. No event data is reachable while not ready. Task 2 asserts GET /graphql → 404 on the probe surface. |
| T-05-05-FC | Denial of service / fail-open | A gate failure leaves the probe server serving indefinitely with `ready=false` instead of exiting | mitigate | The gate chain runs on the main task with `?`-propagation to `main`'s `anyhow::Result<()>`. Any gate `Err` returns from `main` → process exits non-zero; the spawned probe task is dropped on exit. `/ready` never reaches 200 on a failed startup. Fail-closed (V7) is preserved exactly as in 05-01; only the bind/serve ordering changed. |
| T-05-05-AC | Information disclosure | `/ready` exposing internal startup state on the live socket earlier than before | accept | `/ready` exposes only a boolean as an HTTP status code (200/503) — no internal detail, paths, or error text. This is the intended OPS-01 semantics; the CR-01 loopback default + non-loopback warning still govern whether the socket is publicly reachable. Carried forward from T-05-01. |
| T-05-SC | Tampering | npm/pip/cargo installs | mitigate | No new Cargo dependencies (RESEARCH "No new Cargo dependencies required"); uses std `AtomicBool`/`Arc`, existing `tokio` (`spawn`, `sync::Notify`, `with_graceful_shutdown`) and `axum`. No package legitimacy gate needed. |
</threat_model>
