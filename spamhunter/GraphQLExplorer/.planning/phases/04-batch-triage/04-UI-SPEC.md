# Phase 4 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> This is the **fourth and final UI surface** of the v1.0 milestone. Phases 1–3 locked the foundational design system (tokens, type scale, color roles, copy voice) and the load-bearing honesty contracts (non-removable window denominator, asymmetric "absence ≠ clean" framing, escaped-plaintext rendering, accent reserved for the single "go" action). **This phase INHERITS that contract verbatim and EXTENDS it** — it does NOT relitigate spacing, typography, color, font choices, or the design-system tool decision. The effort here is on the TWO new surfaces — a **batch-import view** and a **sortable triage table** — reachable via a NEW `#/batch` hash route, plus the three new honesty contracts those surfaces introduce: the **import summary** (valid / duplicate / unparseable transparency), the **batch window-honesty** denominators ("triaged N of M authors" + "N authors snapshot"), and the **per-author per-signal indicator** treatment (transparent, neutral/amber, never a single opaque score, never "clean"/green).

---

## Surface Scope (Phase 4)

Phase 4 scales the single-suspect workflow to many. It adds the app's **second top-level route** (`#/batch`, alongside the existing dashboard home and the `#/a/<hex>` drill-down) and two new views behind it: a **batch-import view** (paste textarea + file upload + "enumerate corpus" source) and a **sortable triage table**. Clicking a triage row reuses the EXISTING `#/a/<hex>` drill-down (no new drill-down code, no new route for it). The app remains a read-only forensic/utilitarian tool for a single local analyst.

The honesty posture is, again, load-bearing — applied at TWO new scales this phase:

- **The author set is a window.** The enumerated corpus is a *live snapshot* ("N distinct authors as of this fetch"), and a partially-triaged batch shows "triaged N of M authors" — the window-honesty denominator generalized from per-author events to the author set itself.
- **The triage table is a FIRST-PASS SCREEN, never exoneration.** Every indicator is computed over a deliberately tiny per-author window (`perAuthor=5`), so a quiet row means "nothing surfaced in 5 events," NOT "clean." There is **no opaque spam score** and **no `clean`/`ok`/`safe` verdict column** — the asymmetry carried forward from Phases 2–3.

New surfaces this contract covers:

1. **Batch-import view (BATCH-01, BATCH-04)** — a `--surface` view at `#/batch` with three import sources feeding ONE pipeline: a **paste textarea**, a **file upload** (.txt/.csv drop zone + picker), and an **"enumerate corpus"** action (the `authors`-query loop with a running count + a **Stop** control). After normalize/dedupe it shows an **import summary** ("N valid · M duplicates removed · K unparseable", with the unparseable entries listed — never silently dropped) and a single accent **"Triage"** submit. A **large-set warning** (non-blocking) appears before triaging a very large set.
2. **Sortable triage table (BATCH-02, BATCH-03)** — the results surface. Rows matched **strictly by the `author` key** (never index-zipped). Sortable columns (default: event count descending). Per author, **transparent per-signal indicators** (event count in window, burst flag, near-dup flag, tag fan-out flag) rendered as neutral chips/dots with amber-on-signal — NOT one opaque score, NO "clean" column. Authors with zero matching events shown **explicitly as "0 events"**. Each row click → the existing `#/a/<hex>` drill-down.
3. **Batch progress + partial/error states (BATCH-02)** — the sequential chunked `latestPerAuthor` loop and the `authors` enumeration loop both surface progress (in-progress running count, partial results with retry, per-chunk error-with-retry) using the inherited connecting / recoverable / hard-fail semantics. A failed chunk keeps partial results and offers retry; it NEVER kills the whole batch.
4. **Batch window-honesty indicators (BATCH-02, BATCH-04)** — reuse the existing `WindowIndicator` pattern for two new batch-scale denominators: **"triaged N of M authors"** (how much of the batch has results) and **"N distinct authors as of this fetch"** (the enumerated-snapshot count). A partial batch is never framed as a complete picture.

There are **no destructive user actions** in this phase (the API and tool are read-only). Pasting, uploading, enumerating, triaging, sorting, stopping enumeration, and navigating are all non-destructive reads. (Clearing the import textarea is a trivially-reversible local edit, not a destructive action — no confirmation.)

**Out of scope (do not design here):** server-side / cross-author batch analysis, persisted / saved suspect lists, re-run history, CSV export of triage results, multi-kind simultaneous triage, any new single-author signal (Phase 3), and any change to the inherited design system.

**Phase-4 data rendering rules (extend the mono-for-data discipline):**

- **Author column** renders the pubkey in **mono**, truncated (first/last 8 chars, mirroring the Phase-3 mentioned-pubkey treatment) with the full lowercase-hex available on hover/`title`. The row's drill-in target is this exact normalized hex. (Optionally a derived `npub…` mono on a second line/hover — reuse `parseIdentifier(hex).npub`, the single identifier module, never a second nip19 call site.)
- **Listed unparseable tokens** are author-supplied input: render as **escaped plaintext in mono**, one per line (or comma-joined), each truncated with ellipsis if long. They are shown verbatim-but-escaped so the analyst can see exactly what failed to parse — never executed, never `dangerouslySetInnerHTML`.
- **Per-signal indicator values** (the event count, and any numeric the chips expose) render in **mono**; their **labels** ("burst", "near-dup", "high fan-out", "0 events") render in **sans label**.
- **All counts** (valid / duplicate / unparseable, triaged N of M, enumerated author count, per-author event count) use `Intl.NumberFormat` locale grouping (consistent with Phases 1–3, the same `formatInt` the `WindowIndicator` uses) and render in mono.

**Semantic state colors** (inherited, meaning-bound — NOT decorative). Phase-4 mapping:

| State | Token | Meaning in Phase 4 | Reads as |
|-------|-------|--------------------|----------|
| Connecting / info | `--connecting` #5AA9E6 | the first chunk / first enumeration page gated on `/ready` / `503` backoff (reuses the shared `ConnectingShell`) | "working / warming up" — never an error |
| Recoverable / signal-to-investigate | `--recoverable` #E0A24E | a per-author **tripped indicator** in the table (burst / near-dup / high tag fan-out), and the **"partial window"** segment of either batch window indicator ("triaged N of M" while M < total, "snapshot — enumeration in progress / stopped") | "look here — suspicious / incomplete, not a verdict" |
| Recoverable / retryable | `--recoverable` #E0A24E | a **per-chunk fetch error** that is retryable (`413` shrink-and-retry, `503` warm-up, `INVALID_CURSOR` on the enumeration loop, network blip) — partial results retained, **Retry** offered, batch NOT killed | "self-correcting / clear next step" |
| Hard failure | `--hard-fail` #E5534B | a code-less internal error on a chunk, or the relay is unreachable — shown inline against the affected chunk/rows, generic, never leaking internals; other completed rows remain | "genuine failure — generic, no internals leaked" |

**Triage-table & per-signal-indicator color rules (load-bearing honesty constraints):**

- **No opaque score, no verdict color.** There is **no single "spam score" column** and **no `clean`/`ok`/`safe` column**. Each signal is its own transparent column. A signal that is **not** tripped renders **neutral muted** (a dash, a muted "—", or a quiet "none") — it MUST NOT render green/teal and MUST NOT read as "clean."
- **Amber-on-signal only.** A **tripped** indicator (burst detected, near-dup cluster present, tag fan-out over threshold) is tinted **`--recoverable` (amber)** as a chip/dot, ALWAYS paired with a text label ("burst" / "near-dup" / "high fan-out") and a shape (dot/chip) — color is never the sole signal. Amber means "a pattern surfaced in this tiny window — investigate," never a proven verdict.
- **"0 events" is data, not absence.** An author whose triage window returned zero matching events renders an explicit muted **"0 events"** cell (mono count `0` + sans "events" / a muted "0 events" chip) — visually distinct from a quiet-but-non-empty row. It is neither amber nor green: a zero-match author is a *fact about the corpus window*, never an omission and never "clean."
- **The event-count column is neutral.** The per-author event count (0–`perAuthor`) is neutral mono data; a higher count is NOT "worse" colored — it is just the denominator the other signals were computed over. No heatmap, no red/green gradient.
- **No green, no success color, no teal** anywhere in the triage table or its indicators.

**Import-summary & enumeration color rules:**

- The import-summary counts are **neutral data**: "valid" neutral, "duplicates removed" neutral muted, "unparseable" rendered as a **neutral-with-amber-emphasis** segment (amber dot/label) because unparseable input is a *thing the analyst should see and act on* (fix the list), with the offending tokens listed — never silently dropped, never red (it is not a hard failure), never green.
- The **large-set warning** before triaging is **amber** (recoverable tone — a caution the analyst consciously overrides), paired with a label + the count, NOT red and NOT a hard block.
- The **enumeration running count** is neutral mono; while running it carries the **connecting/working** tone (info-blue dot or a neutral "enumerating…" affordance), and the snapshot, once complete or stopped, is framed by its window indicator (amber "snapshot — incomplete" if Stop was pressed before `hasMore` went false).

Destructive: **none** — no destructive user actions (read-only tool). No destructive-confirmation color is introduced.

Color is never the sole carrier of meaning: every tripped indicator, "0 events" cell, partial-batch / snapshot segment, unparseable-count flag, large-set warning, and chunk error pairs its color with a text label and/or a shape (dot, chip, inline message).

## State Treatments (Phase 4 interaction contract)

Each state is **distinct and non-blank**. The highest-stakes treatments are the **import summary** (valid vs duplicate vs unparseable must never blur), the **per-signal indicators** (a quiet row must never read as "clean"), the **"0 events" row** (data, not omission), and the **two batch window denominators** (a partial batch / a stopped snapshot must never read as the whole picture).

### Triage-table honesty framing (BATCH-03 — the central honesty contract, persistent)

| Property | Contract |
|----------|----------|
| **First-pass-screen framing (persistent)** | A muted body line above/below the table, **always rendered, non-dismissible**: "This table is a first-pass screen over the latest {perAuthor} events per author — quiet rows are not cleared authors.
