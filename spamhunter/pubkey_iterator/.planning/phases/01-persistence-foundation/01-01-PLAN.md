## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| pipeline (first-party) → SQLite write API | Pubkey hex strings, layer names, and numeric sub-scores cross into bound SQL parameters. Data originates from the project's own pipeline (not an external request), but is still bound, never interpolated. |
| DB file ↔ OS filesystem | `spamhunter.sqlite` + `-wal`/`-shm` sidecars on the local filesystem; durability governed by WAL + `synchronous=NORMAL`. |

**No network listener, no auth/session/access-control surface, and no untrusted external request boundary exist in this phase** — it is a local, read-only-upstream batch engine writing a local SQLite file. Web/auth/session threats are explicitly out of scope (stated rather than invented).

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | All writes in `writer.rs` + `begin_run`/inline test SQL | mitigate | SQL injection via string-built queries: bind EVERY value with `?N` placeholders + `params![]`; NEVER `format!` a pubkey, layer name, or value into SQL. rusqlite binds parameters even for first-party data.
