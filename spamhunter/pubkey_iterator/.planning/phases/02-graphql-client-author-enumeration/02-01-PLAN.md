<objective>
Extend the Phase-1 single-writer store with the run-state read/update API and the pubkey-only idempotent insert path that the enumerator (plan 02) persists through. Phase 1 only built `begin_run` + the score/signal write path; the resume/abort/drift/pubkey-enumeration helpers do not exist yet.

Purpose: Resumability (D-01/D-02/D-03), pubkey persistence (D-04/D-05), abort-with-preserved-cursor (D-06/D-07), and the drift probe (D-09) all depend on `run`-state helpers and a pubkey-only write that respects the load-bearing single-writer invariant. This plan creates exactly that surface, reusing the existing `run` schema (no migration) and the existing `UPSERT_PUBKEY` const.

Output: A widened writer-channel payload (`WriteMsg` enum), a pubkey-batch insert path through the single writer, and six `run`-state helpers on `Store`, all unit-tested.

Resolves RESEARCH Open Q1 / A5: **route pubkey inserts AND run-state updates through the single writer actor as ordered messages** (the enum-message refactor), NOT short-lived write connections. Rationale: the batched actor (`BATCH=8192`) can otherwise let a `set_run_cursor` on a short-lived connection commit BEFORE the writer flushes the pubkey batch — advancing the cursor past unwritten pubkeys (RESEARCH Pitfall 2). Routing both through one ordered channel guarantees a cursor write is enqueued strictly after the pubkeys of that page, and keeps Phase 3's fetch payload additive (D-11).
</objective>
