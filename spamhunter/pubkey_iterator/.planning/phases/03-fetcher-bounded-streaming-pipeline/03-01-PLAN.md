<objective>
Build the fetch slice of the bounded-memory pipeline: the additive `latestPerAuthor`
query on the existing Phase-2 GraphQL client, the author→pubkey matching that defuses
the INGEST-04 zip-by-index landmine, the 413 shrink-and-retry batch policy, and the
read of the persisted `pubkey` table that is the pipeline's enumeration source (D-07).

This is the data-layer slice Plan 02's pipeline consumes. After this plan a developer can,
in a unit test, hand a list of requested pubkeys to `fetch_batch` against a loopback stub
and get back per-author event groups correctly keyed on `author`, with an over-large batch
transparently split — proving the fetch path end-to-end without the channel wiring.

Purpose: lock the fetch contract (query shape, match-by-author, 413 handling) so Plan 02
wires a proven fetcher into the bounded channel rather than co-developing both at once.
Output: `latestPerAuthor` query + structs + client wrapper, `src/fetch.rs` (fetch_batch +
match_groups), `read_pubkeys` store helper, module exports.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| pubkey table → fetch | Pubkeys read from our own Phase-2 `pubkey` table — already validated at the Phase-2 trust boundary (`enumerate::is_valid_pubkey`); trusted input here. |
| adapter response → structs | The LMDB2GraphQL adapter's `latestPerAuthor` JSON crosses into our typed structs. `kind`/`createdAt` are adapter-claimed integers; `content`/`tags` are untrusted text but this phase does not interpret them (no-op consumer in Plan 02; Phase 4 owns content validation). |
| operator URL → reqwest | The endpoint is operator-supplied (`LMDB2GRAPHQL_URL` / `GraphQlClient::new`), not user input — same posture as Phase 2. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | Event.kind / Event.created_at deserialization | mitigate | Decode as `i64` per contract §8 (64-bit); using i32 would silently truncate adapter-claimed values. Enforced by the queries.rs struct types (Task 1) and grep-gated in acceptance. |
| T-03-02 | Tampering | latest_per_author query construction | mitigate | The `authors` list is passed as a GraphQL **variable** via `json!`, never string-interpolated into the query document (parameterization analog to SQL `?N`). |
| T-03-03 | Denial of Service | oversized author batch / response | mitigate | `fetch_batch` shrinks-and-retries on 413 (D-02) and the batcher (Plan 02) chunks at ≤1000 (contract §12), so a single call never demands an unbounded response. The full bounded-memory control lives in Plan 02's channel. |
| T-03-04 | Information Disclosure | error logging | mitigate | Reuse the Phase-2 `ClientError` discipline — it carries only status + first message, never the full response body (`client.rs:26-28`). No new error type; no body logging added. |
| T-03-05 | Tampering | operator-supplied endpoint (SSRF surface) | accept | Endpoint is operator config, not user input (same as Phase 2 T-02-10). SSRF hardening is a forward OPS-03 concern, not a Phase-3 blocker. |
| T-03-SC | Tampering | `cargo add rayon` (Plan 02) | mitigate | No install in this plan. The rayon add happens in Plan 02; RESEARCH §"Package Legitimacy Audit" verdict: rayon OK (7.0M/wk, github.com/rayon-rs/rayon, since 2015). |
</threat_model>
