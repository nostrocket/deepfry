<objective>
Wire the bounded-memory streaming pipeline that is the structural heart of the phase
(INGEST-03): a tokio fetcher batches the enumerated pubkeys and calls the Plan-01
`fetch_batch`, pushing `AuthorGroup`s into a **bounded** `flume` channel; a rayon-side
drain loop consumes them through an **injected closure** (Phase 3: a no-op counter — D-06).
The bounded channel is the back-pressure point, so peak in-flight memory is capped by
channel capacity, never the corpus size.

This is the slice that makes the whole vertical path real: after this plan, a developer can
run `run_pipeline` over a synthetic million-author set behind a mock fetcher and watch the
producer throttle to the consumer's pace with a provably-bounded in-flight watermark — the
headline D-08 proof — and the no-op count equals the synthetic total (nothing dropped).
The same `run_pipeline` against the live adapter proves `latestPerAuthor` deserializes for real.

Purpose: lock the async↔sync concurrency boundary and the Phase-4 consumer seam before any
detection layer depends on it (CONTEXT phase boundary).
Output: `rayon` dep, `src/pipeline.rs` (run_pipeline + no-op consumer + the watermark /
end-to-end / live tests), module export.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| fast fetcher → slow consumer | The producer can outrun the consumer; without a bound this exhausts memory. The bounded channel is the control. |
| adapter response → pipeline | latestPerAuthor groups cross from the adapter into in-flight buffers; bounded by channel capacity. Content is not interpreted this phase. |
| operator URL → live test | The live D-09 endpoint is operator/CONTEXT-supplied, not user input. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-06 | Denial of Service | unbounded in-flight buffering (fast fetch / slow consume) | mitigate | `flume::bounded(channel_cap)` — `send_async` back-pressures the fetcher when the channel is full, so peak in-flight memory is bounded by capacity, not corpus size (INGEST-03). Proven by the `bounded_memory_watermark` test asserting watermark ≤ cap + batch. |
| T-03-07 | Denial of Service | per-call cost (authors × perAuthor index scans) | mitigate | `authors_per_call = 250` (≤1000 cap, contract §12) and `perAuthor = 100` (≤500 clamp) keep any single call bounded; the 413 split (Plan 01) is the safety net. |
| T-03-08 | Tampering / reactor starvation | async↔sync boundary misuse | mitigate | Producer uses `send_async().await` (never blocking send on a tokio worker); consumer `recv()` runs on a std::thread/rayon worker (never on a tokio worker). Grep-gated in acceptance. |
| T-03-09 | Information Disclosure | live-test logging | mitigate | The live D-09 test logs only a status/skip note, never full response bodies (Phase-2 ClientError discipline carries status + first message only). |
| T-03-SC | Tampering | `cargo add rayon@1.12` | mitigate | RESEARCH §"Package Legitimacy Audit": rayon verdict OK (crates.io, 7.0M/wk, github.com/rayon-rs/rayon, since 2015-12); no postinstall scripts (N/A for crates). No `[ASSUMED]`/`[SUS]` package → no blocking-human checkpoint required. |
</threat_model>
