<objective>
The `run` slice ŌĆö the headline MD-02 fix. Lift the exact scoring wiring that
today lives only in `pipeline.rs` `#[cfg(test)]` (the
`stage.score(run_id, author, events, whitelisted)` ŌåÆ `store.persist(p)` consumer
driven by `run_pipeline` + `production_fetch_with_whitelist`) into a production
`src/run.rs` `run_batch`, and expose it behind a real `clap` CLI in `src/main.rs`.

full end-to-end batch: enumerate the corpus ŌåÆ fetch ~100 events/pubkey through
the bounded pipeline ŌåÆ score via the layers+combiner ŌåÆ persist score/signal ŌĆö
all on ONE run_id carrying the Žä + weight snapshot ŌĆö with a stderr progress
counter to completion. This is the first task after which a real user can DO
something they could not before (the deliverable actually runs).

Purpose: Make scoring reachable in production (not only from tests) on a single
canonical run (Plan 01's lifecycle change), with the reproducibility snapshot
recorded (D-04/D-06).
Output: `src/run.rs` (`run_batch`, `RunError`), the full clap CLI in
`src/main.rs`, `pub mod run` in lib.rs.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator ŌåÆ CLI args | `--config`/`--db`/`--resume`/`--run-id` parsed+typed by clap (V5 input validation) |
| run_batch ŌåÆ adapter (:8080) | untrusted-network read; events fetched, never written back (read-only by design) |
| run_batch ŌåÆ whitelist (:8081) | untrusted-network read; L0 membership only, fails toward not-flagging on outage |
| consumer thread ŌåÆ SQLite | score/signal persisted via the single-writer actor; `'static` Arc<Store> bound |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-04 | Tampering | clap arg parsing (`--db`/`--config` paths) | accept | operator-supplied local paths are trusted (local single-operator CLI; no untrusted-network path input) ŌĆö RESEARCH ┬¦Security V5 |
| T-05-05 | Denial of Service | whitelist outage during `run` | mitigate | `production_fetch_with_whitelist` / `WhitelistClient` already fails toward NOT-whitelisted (no mass false-positive) ŌĆö inherited T-04-07 |
| T-05-06 | Repudiation | reproducibility of a verdict | mitigate | run_batch snapshots Žä+weights into `run.config_json` before scoring (D-04/D-06); the scored run is the canonical run (one run_id) |
| T-05-07 | Information Disclosure | logging event bodies in progress output | mitigate | progress is a count-only `eprintln!` ("scored N pubkeys") ŌĆö never logs event content (don't-log-full-bodies note) |
| T-05-08 | Tampering | consumer running HTTP/scoring on the tokio reactor | mitigate | run_pipeline isolates the consumer on a std::thread (Pitfall 4); fetch+L0 stay in the tokio stage (Pitfall 5) ŌĆö inherited invariants, not re-violated |
</threat_model>
