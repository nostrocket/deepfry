<objective>
The heart of the correctability loop: a `tune` subcommand (TUNE-02) that fits a
logistic model with linfa-logistic over the stored `signal × backpropagation`
join, then — before adopting anything — backtests the freshly-fit weights against
the full labeled set (TUNE-05, D-06). New weights are written to the `weight`
table with provenance ONLY if the backtest passes; a regression (any new false
negative OR any new false positive) blocks adoption and is a pure no-op on the
live `weight` rows (the FP-averse default the project's dominant risk demands).

Resolves RESEARCH Open Q1: τ (`_threshold`) is NOT re-fit — logistic regression
produces layer weights + bias only; `_threshold` is left untouched. Resolves Open
Q2: the gate is STRICT — zero new FN AND zero new FP (D-06's absolute reading).

Purpose: deliver the complete, safe re-tuning mechanism in one slice — fit AND
its safety interlock together, never fit-without-gate.
Output: src/tune.rs (run_tune + backtest + weight write), the Tune clap arm, and
optional config hyperparameter fields.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| backpropagation/signal rows → tune SQL | The fit reads labels + sub-scores; values bound into SQL. |
| fitted weights → weight table | New weights are adopted into live scoring only through the backtest gate. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-02 | Tampering | tune SQL (join read + weight UPDATE) | mitigate | Every value bound with `params![]`/`?N`; the join SELECT + the weight UPDATE interpolate nothing (T-05/V5 discipline, matching writer.rs/export.rs). |
| T-06-03 | Tampering (verdict integrity) | a bad fit adopting harmful weights | mitigate | The backtest gate (TUNE-05/D-06) is the control: staging weights are written ONLY on a strict PASS (zero new FN, zero new FP); a regression blocks and is reported, old weights stay in force. This is the safety interlock protecting the suspected-spammer list's integrity. |
| T-06-04 | Information disclosure | tune report / logging | accept | The tuner reads only sub-scores + labels (numbers + pubkeys); the report names pubkeys + counts, never event content (consistent with the Phase-5 T-05-07 posture). Operator-trusted local CLI. |
| T-06-05 | Tampering | label poisoning (wrong operator labels) | accept | Out of trust scope (operator trusted, ASVS L1); `backpropagation.source`/`note` retain a forensic audit trail. |
</threat_model>
