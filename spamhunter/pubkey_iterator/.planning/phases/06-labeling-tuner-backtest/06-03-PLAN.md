<objective>
Two refinements that complete the correctability loop's honesty + traceability:

TUNE-04 (D-05): the `review_queue` negative-sampling table — a deterministic
random sample of UNFLAGGED pubkeys surfaced for human review, so labels aren't
drawn only from the flagged export (selection bias / RESEARCH Pitfall 2). Without
it, the logistic fit learns "everything is spam" and the backtest trivially
passes while live FPs explode.

TUNE-03 (D-04): each run already reads the latest weights at startup and snapshots
them into `run.config_json` (RESEARCH confirms this is implemented in run_batch).
This plan adds the CONFIRMING test that a *retuned* weight set propagates into the
next run's snapshot — so any past score is traceable to the exact weights that
produced it.

Purpose: keep the training set honest (real negatives) and prove reproducibility
end-to-end. Refines the Plan-02 tune slice; adds no new public surface beyond the
review_queue population folded into `run_tune`.
Output: review_queue population in tune.rs + its deterministic test; the TUNE-03
confirming assertion in run.rs.
</objective>

    - After `run_tune` over a run, `review_queue` contains up to `review_sample_size` rows, each an UNFLAGGED pubkey (score.suspected=0) from that run, with its score and a sampled_at timestamp.
    - The selection is deterministic: order unflagged pubkeys by a stable hash of the pubkey string (no RNG, no rand dep) and take the first K — the same run + same sample size → identical rows every time (D-08/OPS-02 spirit).
    - Populating review_queue is idempotent: re-running replaces the run's rows (DELETE the run's review_queue rows then INSERT), never duplicates.
    - review_queue population touches ONLY the review_queue table (single-writer invariant preserved — it is not one of the writer actor's tables).
  </behavior>

Order them deterministically in Rust by a stable hash of the pubkey string — reuse the existing xxh3/xxhash the fingerprint module already uses (so no new `rand` dependency and the order is reproducible across runs). Take the first `review_sample_size` (config `[tune].review_sample_size`, default 100 per RESEARCH A2; Claude's discretion). 

    Write them on a short-lived non-actor conn (mirror `weight_write_conn`; add a `review_queue_write_conn` on Store if a dedicated accessor reads cleaner): in one transaction, `DELETE FROM review_queue WHERE run_id = ?1` then `INSERT INTO review_queue (run_id, pubkey, score, sampled_at) VALUES (?1, ?2, ?3, ?4)` for each sampled pubkey (sampled_at = now epoch secs). Idempotent re-population. All binds parameterized (`params![]`/`?N`, never format! — T-05/V5).
