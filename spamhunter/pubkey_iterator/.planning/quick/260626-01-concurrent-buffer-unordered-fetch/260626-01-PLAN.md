# Quick Task 260626-01: Concurrent buffer_unordered fetch in run_pipeline

## Goal

`run_pipeline`'s producer fetches one batch at a time (`for batch in
pubkeys.chunks(...) { fetch(batch).await }`), so only one GraphQL request is ever
in flight. Measurements showed the fetch stage is I/O-bound and serial — a single
slow batch stalls the whole pipeline. Overlap the network round-trips by running
up to N batch fetches concurrently while preserving the bounded-memory guarantee
(INGEST-03) via the existing bounded flume channel + `send_async` back-pressure.

## Context (from prior measurements)

- Adapter `latestPerAuthor` is ~0.15–0.85s per 250-author batch, run strictly
  serially today → aggregate ~2.3s for 2000 authors.
- CPU is negligible (consumer is not the bottleneck) — this is a network-latency win.
- The bounded-memory watermark proof (`bounded_memory_watermark`) currently asserts
  `peak <= channel_cap + authors_per_call`; concurrency widens the constant to
  `channel_cap + (concurrency + 1) * authors_per_call` — STILL O(1) in corpus size,
  so INGEST-03 (bound independent of author-set size) holds.

## Approach

Keep `run_pipeline`'s public signature stable (avoids touching 7 call sites +
`run.rs` + `run_pipeline_noop`). Drive concurrency from a new documented module
const so it stays empirically tunable like `DEFAULT_CHANNEL_CAP`/`DEFAULT_AUTHORS_PER_CALL`.

## Tasks

1. **Add `futures-util` dependency** (`Cargo.toml`). It is already in the lockfile
   transitively (via reqwest); promote it to a direct dep for `StreamExt` /
   `stream::iter` / `buffer_unordered`. Pin `0.3`.

2. **Rewrite the producer in `run_pipeline`** (`src/pipeline.rs`):
   - Add `pub const DEFAULT_FETCH_CONCURRENCY: usize = 8;` with a doc comment.
   - Replace the serial `for batch in chunks` loop with
     `stream::iter(chunks.map(|b| fetch(b.to_vec()))).buffer_unordered(DEFAULT_FETCH_CONCURRENCY)`,
     draining each completed batch's `ScoredInput`s into the bounded channel via
     `tx.send_async().await` (back-pressure preserved — while we await a send the
     stream isn't polled, so a full channel pauses new fetches).
   - Preserve error semantics: a fetch error short-circuits (`result?`); a send
     error (consumer gone) stops fetching with `Ok(())`. Still `drop(tx)` + join.
   - Update the module/producer doc comments to describe the concurrency.

3. **Update the bounded-memory proof** (`bounded_memory_watermark` test): assert the
   widened-but-still-constant bound `peak <= CAP + (DEFAULT_FETCH_CONCURRENCY + 1) * APC`
   and keep the non-vacuous `peak > CAP` check.

## Verification

- `cargo build --release` clean.
- `cargo test --lib` — all pass, especially `bounded_memory_watermark`,
  `pipeline_endtoend_count`, `rerun_endtoend_is_deterministic`.
- Live re-measurement: `run --limit 2000` against the live adapter — confirm the
  aggregate fetch time drops and no regression in scored count / determinism.

## Out of scope

- Per-call concurrency parameter (const is sufficient for v1; signature stays stable).
- Parallel scoring consumer (measured CPU-negligible — not worth it).
