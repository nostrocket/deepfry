# Quick Task 260626-01 Summary: Concurrent buffer_unordered fetch

## What changed

`run_pipeline`'s producer no longer fetches one batch at a time. It now drives up
to `DEFAULT_FETCH_CONCURRENCY` batch fetches concurrently via
`futures_util::StreamExt::buffer_unordered`, draining each completed batch's
`ScoredInput`s into the existing bounded flume channel as it arrives. The bounded
channel + `send_async().await` still provide back-pressure (while a send awaits,
the fetch stream is not polled, so a full channel pauses new fetches).

- `Cargo.toml`: promoted `futures-util` (already transitive via reqwest) to a
  direct dep (`default-features = false`, `std` + `async-await`).
- `src/pipeline.rs`:
  - Added `pub const DEFAULT_FETCH_CONCURRENCY` (see tuning note below).
  - Rewrote the producer loop to `stream::iter(chunks).buffer_unordered(FC)`.
  - Preserved error semantics: a fetch error short-circuits (`result?`); a send
    error (consumer gone) stops with `Ok(())`; `drop(tx)` + join unchanged.
  - Updated module + producer doc comments.
  - Updated the `bounded_memory_watermark` proof to the new (still O(1) in corpus)
    bound `peak <= channel_cap + (DEFAULT_FETCH_CONCURRENCY + 1) * authors_per_call`.

## Concurrency set conservatively to FC=2

Set `DEFAULT_FETCH_CONCURRENCY = 2` (not 8) on the user's call. Rationale: during
tuning the LMDB2GraphQL adapter entered a crash-loop under load, and this project
has a documented `MDB_BAD_RSLOT` history (LMDB reused-read-slot crashes under
concurrent reads). FC=2 still overlaps the serialized network latency without
firing a large burst of concurrent multi-MB LMDB reads. `FC=1` reproduces the
original strictly-serial behavior; raise once the adapter is verified safe under
concurrent reads. The const is the single tuning knob (no signature change, so no
call-site churn).
