# Phase 01.1 Plan 02: GoBridgeTransport Binary-Stream Decode Summary

**A third `GraphTransport` that fetches the Go bridge's binary frame at `/graph.bin` (same-origin via a Vite proxy) and decodes it incrementally with `fetch` + `ReadableStream` into SoA typed arrays with ZERO `JSON.parse` — the PERF-01 drop-in swap that removes the single-shot memory-doubling parse wall behind the unchanged `GraphTransport` seam.**

## Accomplishments

- **`wireFormat.ts`** — the single TS source of frame-layout truth, mirroring `bridge/internal/wire/encode.go` exactly: `MAGIC = 0x42544f57` ("WOTB" LE, == Go `MagicWOTB`), `VERSION = 1`, `HEADER_BYTES = 16`, `PUBKEY_BYTES = 32`, and a pure `decodeFrame(buffer)` that reads the LE header with a `DataView`, throws on MAGIC/VERSION mismatch (T-01.1-04) *and* on a truncated body, then exposes `Uint32Array`/`Int32Array` VIEWS over every section in wire order (edges, inDeg, outDeg, community u32; kind3CreatedAt, lastDbUpdate i32; packed 32-byte pubkeys last).
- **`GoBridgeTransport`** — implements `GraphTransport.load(onProgress)`: `fetch('/graph.bin')` → `getReader()` loop accumulating `Uint8Array` chunks and emitting `stage:'receive'` (byte-derived edge estimate, Assumption A5), then a single copy into one offset-0 `ArrayBuffer`, `decodeFrame`, `buildGraphBuffers` (keeping the MAX_NODE_INDEX 2^24 Float32 precision assertion), hex-encoding of the packed pubkeys into `hexByIndex`, mulberry32 position seeding, and a final `stage:'layout'`. Zero `JSON.parse` in source.
- **Additive `GraphBuffers` extension** (D-04): `outDegree?`, `community?`, `kind3CreatedAt?`, `lastDbUpdate?` — all optional so `SyntheticTransport`/`DgraphTransport` keep compiling; `LoadProgress.stage` gained `'receive'` without removing `'parse'`.
- **Same-origin wiring**: `vite.config.ts` proxies `/graph.bin` to `127.0.0.1:8081` (`changeOrigin`), dodging COEP/CORP entirely while retaining the COOP/COEP `require-corp` headers the Wave 3 heap verdict depends on; `main.ts selectTransport` adds the `?transport=bridge` branch.
- **Round-trip tests**: a synthetic-frame builder mirroring the Go layout drives 6 decode tests (counts, Float32 link pairs, every attribute array, `hexByIndex`, position seeding, `receive`→`layout` progress, MAGIC-mismatch throw) against a stubbed `fetch`/`ReadableStream` — no running bridge required.

## Decisions Made

- **One offset-0 copy before viewing.** Concatenating `ReadableStream` chunks yields a `Uint8Array` whose backing buffer offset is not guaranteed 4-byte aligned, and `new Uint32Array(buffer, byteOffset, n)` throws on a misaligned offset (Pitfall 4). Copying the accumulated bytes into a fresh `new Uint8Array(received)` guarantees offset-0 alignment for all u32 views. This is the one transient copy the RESEARCH "accumulate-then-view" pattern accepts — still far below a `JSON.parse` doubling. A streaming-into-preallocated-buffer optimization is deferred to the Plan 03 heap verdict.
- **Bridge path is not flagged `isDgraph`.** main.ts's `isDgraph` branch casts the transport to `DgraphTransport` to read `lastEncodingNs` for the verdict readout — the bridge doesn't provide that, and the bridge-specific verdict is Plan 03's job. Selecting bridge renders via the existing path without regressing the dgraph verdict wiring.
- **decodeFrame truncation guard.** Beyond MAGIC/VERSION, `decodeFrame` checks the declared `nodeCount`/`edgeCount` fit the buffer length and throws a `RangeError` otherwise, so a short body fails loudly rather than producing an out-of-range typed-array view.
