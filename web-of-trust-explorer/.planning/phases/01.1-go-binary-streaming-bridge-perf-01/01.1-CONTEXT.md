# Phase 01.1: Go Binary-Streaming Bridge (PERF-01) - Context

**Gathered:** 2026-06-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the browser-direct Dgraph JSON wire — which **FAILED** the Phase 1 feasibility
verdict (the real dev DB: 365,559 follow-source nodes / 1,541,632 profiles / tens of
millions of edges drove the M3 Pro into swap on a single-shot, memory-doubling
`JSON.parse`) — with a thin **Go service** that reads Dgraph over **dgo gRPC**, does the
**hex→uint32 remap server-side**, and streams a **binary** buffer to the browser with
**zero `JSON.parse`**. It drops in as `GoBridgeTransport` behind the existing
`GraphTransport` interface (`load(onProgress) → GraphBuffers`); cosmos.gl + the SoA
render/layout pipeline stay untouched.

In scope:
- A standalone Go bridge binary living **inside `web-of-trust-explorer/`**, launched via a
  `make` target, connecting to Dgraph gRPC at `localhost:9080` (read-only).
- Server-side **hex→uint32 dense remap** + the full server-side **data-prep pass**:
  in-degree (followers), out-degree (follows), community ID (Leiden if a solid Go lib
  exists, else Louvain), and per-node activity timestamps (`kind3CreatedAt`,
  `last_db_update`).
- A **binary edge buffer + node-attribute table** streamed over **HTTP chunked**
  (`fetch` + `ReadableStream`) and decoded straight into typed buffers in the browser.
- A new `GoBridgeTransport` implementing `GraphTransport`, swapped in for `DgraphTransport`.
- Loader adapted to the binary path (bytes-received / decode counter) + a **fresh recorded
  feasibility verdict** confirming PERF-01 PASS vs the old JSON FAIL.

Out of scope (other phases): the **visual encoding** of the precomputed attributes —
node sizing/coloring by degree (OVER-01) and community coloring (OVER-02) is Phase 2; the
time-range filter UI that consumes the timestamps (FILT-01), search, node-detail panel,
neighborhood highlight, refresh are Phase 3. This phase ships the **data**, those phases
**render/interact** over it.

Requirements: **PERF-01** (primary). Re-validates the at-scale load capability that
DATA-01 / DATA-03 deferred to this phase via the 01-03 FAIL verdict.

</domain>

<decisions>
## Implementation Decisions

### Run Model & Code Home
- **D-01:** The bridge is a **standalone Go binary** launched by a `make` target (e.g.
  `make build` + `make run` / `go run`), **not** a docker-compose service. The dev starts
  it separately; it connects to `localhost:9080`. Lightest weight, no compose edits.
- **D-02:** Bridge code lives **inside `web-of-trust-explorer/`** (e.g. a Go module under
  `web-of-trust-explorer/bridge/`), keeping the whole tool self-contained in one subdir
  rather than as a new top-level deepfry subsystem. (It still follows the per-subsystem
  Go-module + Makefile convention of the wider stack.)

### Bridge Responsibility Boundary (full server-side data-prep)
- **D-03:** The bridge is **not pure transport** — it is the **single server-side data-prep
  layer**. It packs precomputed per-node attributes into the binary stream so Phases 2 & 3
  consume ready arrays instead of computing client-side.
- **D-04:** Attributes packed into the stream: **in-degree (followers)**, **out-degree
  (follows)**, **community ID**, and **activity timestamps** (`kind3CreatedAt`,
  `last_db_update`) — plus the edge pairs and the index↔pubkey node table.
- **D-05:** Community detection uses **Leiden** server-side **if a production-grade Go lib
  exists**; otherwise **fall back to Louvain (Go)**. Researcher must confirm Go library
  availability and record the actual choice. (Leiden = higher quality, fewer disconnected
  communities — the stack doc's stated server-side win.)
- **D-06:** **Cross-phase impact (MUST propagate to roadmap at planning/transition):**
  moving degree + community + timestamps server-side means OVER-01, OVER-02, and FILT-01
  no longer *compute* their inputs client-side — Phase 2 becomes pure style-buffer encoding
  over bridge-provided arrays, and Phase 3's time filter reads bridge-provided timestamps.
  The client Louvain/in-degree Worker path designed in Phase 1/Phase 2 is **superseded** by
  the server-side pass. Planner/transition must update ROADMAP.md Phase 2/3 success criteria
  and REQUIREMENTS traceability accordingly.
- **D-07:** Because community detection needs the **whole graph**, the bridge does
  **fetch-all-from-Dgraph → compute attrs → stream**. The streaming win is **zero browser
  `JSON.parse` on receive** (and progressive byte decode), NOT server-side incrementality.

### Wire Mechanism & Loader/Verdict
- **D-08:** Binary reaches the browser over **HTTP chunked** (`fetch` + `ReadableStream`),
  decoded chunk-by-chunk into the typed buffers. No WebSocket — one-shot bulk load needs no
  bidirectional channel.
- **D-09:** The loader/verdict instrument (Phase 1 D-09/D-10) **adapts to the binary path**:
  stages become **Fetching → Receiving bytes (MB/s) → Building layout**, with a live
  bytes/edges-decoded counter (no "Parsing JSON" stage). On completion it records a **NEW
  feasibility verdict** — server fetch+compute ms / stream ms / client decode ms / node+edge
  counts / peak JS heap — to confirm **PERF-01 PASS** vs the recorded 01-03 JSON FAIL.

### Claude's Discretion
- **PASS bar / re-verdict target:** user opted not to set an explicit numeric bar (the old
  JSON bar was D-05 ≤~30s). Planner sets a sensible target — **usable on the real dev DB,
  no swap, render still holds 60fps after load**; record the measured numbers as the verdict.
- **Binary format details:** framing/layout of the edge buffer + node-attribute table
  (length-prefixing, attribute packing order, endianness, whether pubkeys ship as packed
  32-byte binary vs hex strings), chunk sizes, gzip/compression on the HTTP body.
- **`GraphBuffers` / `GraphTransport` extension:** `GraphBuffers` (in `src/types.ts`)
  currently has `positions, links, nodeCount, edgeCount, inDegree?, hexByIndex?` — extend it
  for out-degree, community, and timestamps (planner's shape).
- **dgo version / Go module layout / Makefile targets** — follow existing deepfry Go
  conventions (`dgo/v210` is used by web-of-trust).
- **Server-side memory handling** of the full graph (remap map, attribute arrays) during the
  fetch-compute-stream cycle.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `GraphTransport` interface + `createHexRemap()` + `buildGraphBuffers()` +
  `parseDqlEnvelope()`/`extractEncodingNs()` (`src/transport/GraphTransport.ts`) — the
  remap and buffer-build logic to port to Go (server-side), and the interface
  `GoBridgeTransport` implements unchanged.
- `SyntheticTransport` + `DgraphTransport` — two existing `GraphTransport` implementations;
  `GoBridgeTransport` is the third, selectable behind the same seam.
- `src/ui/loader.ts` + `src/ui/verdict.ts` — staged loader + verdict readout to adapt
  (D-09).
- The wider deepfry stack's **web-of-trust** subsystem uses `dgo/v210` against the same
  Dgraph — reference for the gRPC read pattern (read-only).

### Established Patterns
- **Swappable transport seam** (`GraphTransport.load(onProgress) → GraphBuffers`) — the
  whole architecture exists for exactly this swap; cosmos.gl + SoA pipeline are downstream
  of it and untouched.
- **SoA typed buffers, never per-node heap objects** — the binary decode must fill flat
  typed arrays directly (the 18GB-memory discipline from Phase 1).
- **Float32 link-index precision guard** (`MAX_NODE_INDEX` = 2^24) — still applies; node
  count (~1.5M) is safely under it, but `buildGraphBuffers` keeps the assertion.
- **Read-only, never mutate** — grep-verified discipline (0 `/mutate`|`/alter`); the bridge
  must hold the same line over gRPC.

### Integration Points
- Bridge ↔ Dgraph: dgo gRPC `localhost:9080`, read-only.
- Bridge ↔ browser: HTTP chunked binary stream consumed by `GoBridgeTransport` (`fetch` +
  `ReadableStream`), decoded into `GraphBuffers`.
- `GoBridgeTransport` ↔ app: same `GraphTransport` seam wired in `src/main.ts` /
  transport selection.

</code_context>

<specifics>
## Specific Ideas

- The Go bridge is being **promoted** from "pure transport escape hatch" to the **single
  server-side data-preparation layer** for the whole tool — it owns remap + degree +
  community + timestamps. This is a deliberate consolidation, not just a wire swap (D-03,
  D-06).
- The loader remains a **measurement instrument** (continuity from Phase 1's D-09/D-10): the
  binary-path readout exists to record the PERF-01 PASS verdict that closes the 01-03 FAIL.
- "Zero browser `JSON.parse`" is the load-bearing constraint — the entire point of the phase
  is to remove the single-shot memory-doubling parse that put the M3 Pro into swap.

</specifics>

<deferred>
## Deferred Ideas

- **docker-compose service for the bridge** — considered, declined for the standalone
  binary (D-01). Natural escalation if the tool ever needs one-command bring-up alongside
  Dgraph.
- **Server-side caching of the remapped binary buffer** (skip Dgraph re-query / recompute on
  repeat session loads, since a bare count alone is ~6.7s) — not raised by the user; relates
  to INTR-04 (local layout cache, v2). Reconsider if reload time becomes a pain.
- **Bridge serving the single-pubkey search query** (EXPL-01) — Phase 3 territory; keep the
  bridge to bulk-load + data-prep for now.
- **Explicit numeric PASS bar** for the re-verdict — left to planner discretion (see Claude's
  Discretion); revisit if "usable / no swap / 60fps holds" proves too vague to verify.
- **WebSocket transport** — declined for HTTP chunked (D-08); revisit only if a cancel/resume
  control is wanted.

</deferred>
