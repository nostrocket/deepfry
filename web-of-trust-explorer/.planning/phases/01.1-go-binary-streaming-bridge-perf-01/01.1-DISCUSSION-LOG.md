## Bridge responsibility boundary

### What should the Go bridge be responsible for?

| Option | Description | Selected |
|--------|-------------|----------|
| Pure transport (edges + node table) | dgo read → remap → stream edges; client computes degree/community | |
| Transport + in-degree | Also ship server-computed in-degree | |
| Transport + full analytics offload | Also compute degree AND community (Leiden) server-side; ship IDs | ✓ |

**User's choice:** Transport + full analytics offload

### Which precomputed attributes should the bridge pack into the binary stream?

| Option | Description | Selected |
|--------|-------------|----------|
| In-degree (followers) | Drives node sizing/influence (OVER-01) | ✓ |
| Out-degree (follows) | Directed counterpart, distinct encoding (OVER-01) | ✓ |
| Community ID | Drives community coloring (OVER-02) | ✓ |
| Activity timestamps | kind3CreatedAt / last_db_update for time filter (FILT-01) | ✓ |

**User's choice:** All four — in-degree, out-degree, community ID, activity timestamps

### Community detection algorithm in the bridge?

| Option | Description | Selected |
|--------|-------------|----------|
| Leiden (if a solid Go lib exists) | Higher quality; researcher confirms availability, else Louvain | ✓ |
| Louvain (Go) | Safer availability; same algorithm the client would have run | |
| You decide | Researcher picks per mature Go libs | |

**User's choice:** Leiden if a solid Go lib exists (fallback Louvain)
**Notes:** This promotes the bridge from pure transport to the single server-side
data-prep layer, with cross-phase impact on Phase 2 (OVER-01/02) and Phase 3 (FILT-01) —
those become pure client-side encoding/interaction over bridge-provided arrays. Flagged in
CONTEXT.md D-06 for roadmap/requirements update at planning/transition. Community detection
needs the whole graph, so the bridge does fetch-all → compute → stream (D-07).
