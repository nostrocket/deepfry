# Phase 01.1: Go Binary-Streaming Bridge (PERF-01) - Research

**Researched:** 2026-06-23
**Domain:** Go service (dgo gRPC read-all + community detection + binary HTTP streaming) ↔ browser binary decode behind the existing `GraphTransport` seam
**Confidence:** HIGH (codebase + Go conventions verified in-repo; community-detection lib choice MEDIUM)

## Summary

This phase replaces the browser-direct Dgraph JSON wire — which **FAILED** the Phase 1 feasibility verdict (real dev DB: 365,559 follow-source nodes / 1,541,632 profiles / tens of millions of edges drove the M3 Pro into swap on a single-shot, memory-doubling `JSON.parse`) — with a thin standalone **Go bridge** that reads Dgraph over **dgo gRPC**, does the **hex→uint32 dense remap server-side**, runs the **full server-side data-prep pass** (in-degree, out-degree, community ID, two timestamps), and streams a **binary** buffer to the browser over **HTTP chunked** (`fetch` + `ReadableStream`) with **zero browser `JSON.parse`**. It drops in as `GoBridgeTransport` behind the unchanged `GraphTransport` interface; cosmos.gl + the SoA render/layout pipeline are downstream and untouched.

The decisive advantage of this repo is that the **read-all pattern already exists and is proven against the same live Dgraph**: the sibling `web-of-trust` subsystem (`pkg/dgraph/`) reads the whole graph with `dgo/v210` over gRPC using a **UID-cursor (`after:`) paging loop**, `NewReadOnlyTxn()`, and `grpc.MaxCallRecvMsgSize(256 << 20)` to clear the 4 MB gRPC cap. The bridge is a direct application of that pattern with a server-side compute pass appended. Go 1.26.1 is installed; `gonum v0.16.0` is already a resolved dependency in the wider stack.

**Primary recommendation:** Build the bridge as a new Go module under `web-of-trust-explorer/bridge/` mirroring `web-of-trust`'s `dgo/v210` + `NewReadOnlyTxn()` + `after:`-cursor read-all pattern. For community detection (D-05): **there is NO production-grade Go Leiden — fall back to Louvain.** Recommend an **in-bridge array-based Louvain over the dense uint32 CSR edge list** (do NOT route the 1.5M-node graph through gonum's interface-based `graph.Graph` model — see Pitfall 1). Stream a **little-endian, length-prefixed binary frame** (header → edges u32 pairs → per-node attribute table → pubkey table) and decode it incrementally on the browser side with a byte-accumulating `ReadableStream` reader that re-slices on 4-byte boundaries. Set `Cross-Origin-Resource-Policy: cross-origin` + CORS on the bridge or the COEP page will block the fetch (Pitfall 3).

### Pattern 1: Read-only UID-cursor read-all (port of web-of-trust)
**What:** Page the whole `follows` graph by UID cursor, read-only, raising the gRPC receive cap.
**When to use:** The bridge's Dgraph read. This is the **exact proven pattern** from `web-of-trust/pkg/dgraph`.
```go
// Source: web-of-trust/pkg/dgraph/dgraph.go (NewClient, BackfillFollowerCount paging)
const maxRecvMsgSize = 256 << 20 // 256 MiB — clears the 4MB gRPC default cap
conn, _ := grpc.NewClient(addr,
    grpc.WithTransportCredentials(insecure.NewCredentials()),
    grpc.WithDefaultCallOptions(grpc.MaxCallRecvMsgSize(maxRecvMsgSize)))
dg := dgo.NewDgraphClient(api.NewDgraphClient(conn))

// Read-only, UID-cursor paging (after:), NEVER offset (offset degrades; see WoT note).
cursor := "0x0"
for {
    q := fmt.Sprintf(
        `{ q(func: has(follows), first: %d, after: %s) {
             uid follows { uid } kind3CreatedAt last_db_update } }`, pageSize, cursor)
    txn := dg.NewReadOnlyTxn()
    resp, err := txn.Query(ctx, q)   // resp.Json is []byte
    txn.Discard(ctx)                 // INLINE discard inside the loop (not defer)
    // unmarshal page, remap uids, append edges + node attrs, drop the page bytes
    if len(page) < pageSize { break }
    cursor = page[len(page)-1].UID
}
```
**Critical WoT lessons baked in:** (1) `after:` cursor, never `offset:` (offset paging measured ~100 nodes/min at production scale); (2) `txn.Discard(ctx)` **inline** inside the loop, never `defer` inside a loop (defer fires at function return, leaking txns — WoT WR-02/HARD-01); (3) raise `MaxCallRecvMsgSize` or large pages fail with `ResourceExhausted`.

### Pattern 2: Server-side hex→uint32 dense remap (port of `createHexRemap`)
**What:** First sighting of a hex uid assigns the next dense index; repeats return the same index. Identical semantics to `src/transport/GraphTransport.ts createHexRemap()`, just in Go.
```go
// Source: port of src/transport/GraphTransport.ts createHexRemap()
type remap struct{ m map[string]uint32; hex []string }
func (r *remap) indexOf(h string) uint32 {
    if i, ok := r.m[h]; ok { return i }
    i := uint32(len(r.m)); r.m[h] = i; r.hex = append(r.hex, h); return i
}
```
**Note:** Dgraph uids are hex like `0x1a3f`, but the explorer's identity is the **pubkey** (`@id`), not the Dgraph uid. The Phase-1 JSON path remapped on Dgraph `uid`. For the hover tooltip / search (Phase 3), the bridge must also read `pubkey` per node and ship the index→pubkey table (D-04). Decide whether the dense index keys on `uid` (cheaper, smaller) with a parallel pubkey lookup, or directly on `pubkey`. **Recommendation: dense index keyed on uid for edge compactness; ship an index→pubkey string table for the tooltip.**

### Pattern 3: Little-endian length-prefixed binary frame
**What:** A self-describing frame the browser decodes without `JSON.parse`. All multi-byte ints **little-endian** (x86/ARM native; avoids per-element byte-swap in JS — `DataView` defaults big-endian but typed-array views read host LE).
```
[ MAGIC u32 "WOTB" ][ VERSION u32 ]
[ nodeCount u32 ][ edgeCount u32 ]
[ edges:   u32 × (edgeCount*2)   ]   # src,tgt pairs (dense indices)
[ inDeg:   u32 × nodeCount       ]
[ outDeg:  u32 × nodeCount       ]
[ community: u32 × nodeCount     ]
[ kind3CreatedAt:  i32 × nodeCount ] # unix seconds fit i32 until 2038; OR i64/f64 if safety wanted
[ last_db_update:  i32 × nodeCount ]
[ pubkeyBytes: 32 × nodeCount    ]   # packed binary 32-byte pubkeys (NOT 64-char hex → halves this table)
```
**Endianness:** little-endian. A browser `new Uint32Array(buffer, offset, n)` reads host byte order (LE on all target machines) — zero-cost. Only use `DataView` for the header where alignment may not be 4-byte.
**Pubkeys as packed 32-byte binary** (Claude's-discretion item): halves the pubkey table vs 64-char hex and removes hex-decode work; the browser hex-encodes lazily only for the hovered/searched node. **Recommended.**

### Pattern 4: Go chunked binary streaming server
```go
// Source: pkg.go.dev/net/http (Flusher) + Medium "HTTP Chunking ... with Go"
func (s *server) handleGraph(w http.ResponseWriter, r *http.Request) {
    w.Header().Set("Content-Type", "application/octet-stream")
    w.Header().Set("Cross-Origin-Resource-Policy", "cross-origin") // REQUIRED — Pitfall 3
    w.Header().Set("Access-Control-Allow-Origin", "*")             // CORS for the Vite origin
    fl, _ := w.(http.Flusher)
    // Do NOT set Content-Length → Go auto-applies Transfer-Encoding: chunked.
    for _, section := range frameSections {  // header, edges, attrs, pubkeys
        w.Write(section)
        fl.Flush()                            // push each chunk so the browser counter ticks
    }
}
```
**Note:** chunked transfer is HTTP/1.1 only (not HTTP/2). A plain `http.ListenAndServe` on localhost is HTTP/1.1 by default — fine. Flush after each large section so `GoBridgeTransport`'s byte counter advances visibly (D-09).

### Pattern 5: Browser incremental decode (zero JSON.parse)
```ts
// Source: MDN Streams API "Using readable streams" + web.dev typed arrays
const reader = (await fetch(url)).body!.getReader();
let received = 0; const chunks: Uint8Array[] = [];
for (;;) {
  const { done, value } = await reader.read();
  if (done) break;
  chunks.push(value); received += value.byteLength;
  onProgress({ stage: 'receive', edgesSoFar: estimateEdges(received) });
}
const all = concat(chunks);                 // single Uint8Array
const dv = new DataView(all.buffer);
const nodeCount = dv.getUint32(8, true), edgeCount = dv.getUint32(12, true);
let off = 16;
const links = new Uint32Array(all.buffer, off, edgeCount * 2); off += edgeCount*8;
// ... slice inDeg/outDeg/community/timestamps/pubkeys at their byte offsets
```
**4-byte alignment caveat:** `new Uint32Array(buffer, byteOffset, n)` throws if `byteOffset` is not a multiple of 4. Lay out the frame so every u32 section starts 4-byte aligned (the header is 16 bytes; keep section sizes multiples of 4; the 32-byte pubkey table is naturally aligned). The simplest robust path: accumulate ALL bytes then view (one transient copy, but no `JSON.parse` doubling). A streaming-into-preallocated-buffer variant is possible once `nodeCount`/`edgeCount` are known from the header but adds chunk-boundary bookkeeping — **start with accumulate-then-view; optimize only if peak heap is a problem.**

### Anti-Patterns to Avoid
- **Routing the 1.5M-node graph through gonum's `graph.Graph` interface model** — allocates a heap object per node and per edge; reintroduces the exact memory blowup this phase exists to kill. Compute Louvain over flat `[]uint32` arrays.
- **`offset:` paging** — degrades badly at scale (WoT measured ~100 nodes/min). Use `after:` UID cursor.
- **`defer txn.Discard()` inside the paging loop** — leaks transactions until function return (WoT WR-02). Discard inline each iteration.
- **Shipping 64-char hex pubkeys in the stream** — doubles the pubkey table. Pack 32 binary bytes.
- **Big-endian frame** — forces per-element byte-swap in JS. Use little-endian so typed-array views are zero-cost.
- **Forgetting `Cross-Origin-Resource-Policy` on the bridge** — the COEP `require-corp` page silently blocks the cross-origin fetch (Pitfall 3).

## Common Pitfalls

### Pitfall 1: gonum interface-graph memory blowup at 1.5M nodes
**What goes wrong:** Building a `gonum/graph/simple.DirectedGraph` (one `Node` interface value + one `Edge` per relationship) for 1.5M nodes / tens of millions of edges allocates millions of heap objects — the exact failure mode this phase exists to eliminate, just moved server-side. gonum also has a documented infinite-loop bug on small ΔQ (gonum#1488).
**Why it happens:** gonum's API is interface-based and general-purpose, not array-optimized for million-node modularity.
**How to avoid:** Implement Louvain over the dense `[]uint32` CSR edge list directly (community array, weighted-degree array, in-place merges). Use gonum's `louvain_directed.go` only as a correctness reference for directed modularity Q.
**Warning signs:** Bridge RSS climbing into GBs during the prep pass; GC thrash; OOM. [VERIFIED: gonum#1488 + pkg.go.dev community docs note "no scalability guarantees"]

### Pitfall 2: gRPC `ResourceExhausted` on large read pages
**What goes wrong:** A single page response exceeds gRPC's default 4 MB receive cap → `ResourceExhausted`, fatal (non-transient).
**Why it happens:** Hundreds of thousands of `follows{uid}` rows per page easily exceed 4 MB.
**How to avoid:** Set `grpc.WithDefaultCallOptions(grpc.MaxCallRecvMsgSize(256 << 20))` (exactly as WoT does) AND keep page sizes bounded (WoT uses `first: N` windows). [VERIFIED: web-of-trust/pkg/dgraph/dgraph.go:42-47]

### Pitfall 3: COEP page blocks the cross-origin bridge fetch
**What goes wrong:** The Vite app sets `Cross-Origin-Embedder-Policy: require-corp` (for `measureUserAgentSpecificMemory`). A `fetch()` to the Go bridge on a *different port* is cross-origin; under COEP `require-corp` the browser **blocks the response** unless the bridge sends `Cross-Origin-Resource-Policy: cross-origin`. Also needs CORS (`Access-Control-Allow-Origin`).
**Why it happens:** Different port = different origin; COEP requires all subresources to opt in via CORP.
**How to avoid:** Bridge sets `Cross-Origin-Resource-Policy: cross-origin` + `Access-Control-Allow-Origin: <vite origin or *>`. Alternatively, proxy `/graph.bin` through Vite's dev server (`server.proxy`) so it's same-origin (avoids CORS/CORP entirely — **simpler; recommended**). [VERIFIED: vite.config.ts lines 9-12 set COEP require-corp]
**Warning signs:** `fetch` rejects with a CORS/`net::ERR_BLOCKED_BY_RESPONSE` error referencing CORP even though the bridge is reachable via curl.

### Pitfall 4: Uint32Array byteOffset alignment error
**What goes wrong:** `new Uint32Array(buffer, byteOffset, n)` throws `RangeError: start offset ... should be a multiple of 4` if a section doesn't start 4-byte aligned.
**Why it happens:** The 32-byte pubkey table or an odd-length section shifts later sections off a 4-byte boundary.
**How to avoid:** Order sections so all u32 tables come before the 32-byte pubkey table (or pad to 4-byte). Header is 16 bytes (aligned). [CITED: MDN typed arrays — "byteLength must be a multiple of BYTES_PER_ELEMENT"]

### Pitfall 5: dgo/v210 vs Dgraph v25 API surface
**What goes wrong:** Assuming v210 is incompatible with Dgraph v25 and switching to dgo/v240/v250 mid-build, introducing an untested gRPC surface.
**Why it happens:** The proxy/docs say v210 → Dgraph 21.x.
**How to avoid:** The **legacy `api.Dgraph` gRPC surface used by v210 works against the live Dgraph v25.3.0** — `web-of-trust` proves this in production today. Use v210 for parity. Only escalate to v240 if a specific call genuinely fails. [VERIFIED: web-of-trust/go.mod uses v210 against the same DB; STATE.md records Dgraph v25.3.0]

### Pitfall 6: `defer Discard()` inside the paging loop
**What goes wrong:** `defer txn.Discard(ctx)` inside the read loop fires only at function return, accumulating undiscarded read-only transactions across all pages.
**How to avoid:** Call `txn.Discard(ctx)` inline at the end of each iteration. [VERIFIED: web-of-trust WR-02/HARD-01 comments in dgraph.go]

### In-degree + out-degree in one pass over the dense edge list
```go
// Source: port of src/graph/generator.ts computeInDegree, extended for out-degree
inDeg := make([]uint32, nodeCount)
outDeg := make([]uint32, nodeCount)
for i := 0; i < len(edges); i += 2 {
    src, tgt := edges[i], edges[i+1]
    outDeg[src]++  // follows
    inDeg[tgt]++   // followers (D-04) — derived, NOT an inverse-edge query
}
```

### Directed-graph note for in-degree == follower count
The WoT data model stores `follows: [uid] @reverse`; the inverse is `~follows` (followers). In-degree over the `follows` edge list **equals** follower count — so the bridge derives followers in O(E) **without** querying `~follows` (matches the Phase-1 discipline and the `wote-followers-data-model` memory note). ## Open Questions

2. **Dense index key: uid vs pubkey**
   - What we know: Phase-1 keyed on Dgraph `uid`; the tooltip/search needs `pubkey`.
   - What's unclear: whether to remap on uid (compact) + ship a uid→pubkey table, or remap directly on pubkey.
   - Recommendation: remap on uid for edge compactness; read `pubkey` per node and ship an index→pubkey (packed 32-byte) table.

3. **Same-origin proxy vs explicit CORP/CORS**
   - Recommendation: Use Vite `server.proxy` for `/graph.bin` (simplest, dodges COEP/CORP entirely). Keep explicit CORP+CORS on the bridge as the documented production-style fallback.

### Known Threat Patterns for {Go HTTP bridge + dgo gRPC}

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Bridge bound to 0.0.0.0 exposes Dgraph read-all on the LAN | Information Disclosure | Bind the HTTP listener to `127.0.0.1` only |
| Unbounded `first:`/page param → memory exhaustion | Denial of Service | Clamp page size server-side; the bridge controls paging, not the client |
| Accidental write/alter via dgo | Tampering | Use `NewReadOnlyTxn()` exclusively; never construct `api.Mutation`/`Alter` (grep-verify 0 `Mutate`/`Alter` — same discipline as DgraphTransport's `0 /mutate|/alter`) |
| Overly permissive CORS (`*`) on the binary endpoint | Information Disclosure | For a localhost dev tool `*` is acceptable; prefer the exact Vite origin, or use Vite same-origin proxy and drop CORS |
| Malformed/truncated stream corrupts buffers | Tampering | MAGIC+VERSION header + length checks; reject before building typed-array views |

## GraphBuffers / GraphTransport Extension Shape (D-04, Claude's-discretion)

The browser-side contract to extend (`src/types.ts`). Current shape:
```ts
interface GraphBuffers {
  positions: Float32Array; links: Float32Array;
  nodeCount: number; edgeCount: number;
  inDegree?: Uint32Array; hexByIndex?: string[];
}
```
**Recommended additive extension** (all optional so `SyntheticTransport`/`DgraphTransport` keep compiling — non-breaking):
```ts
interface GraphBuffers {
  positions: Float32Array; links: Float32Array;
  nodeCount: number; edgeCount: number;
  inDegree?: Uint32Array;
  outDegree?: Uint32Array;        // NEW (D-04) — follows count
  community?: Uint32Array;        // NEW (D-04) — Louvain community id per node
  kind3CreatedAt?: Int32Array;    // NEW (D-04) — unix seconds (or Float64Array if i64 chosen)
  lastDbUpdate?: Int32Array;      // NEW (D-04)
  hexByIndex?: string[];          // existing — now populated from the packed pubkey table
}
```
`GoBridgeTransport implements GraphTransport.load(onProgress) → GraphBuffers` exactly like `DgraphTransport`; it fetches the binary frame, fills these arrays from the decoded views, seeds positions (same `seedPositions` approach — the wire carries no coordinates), and resolves. `buildGraphBuffers` + the `MAX_NODE_INDEX` 2^24 guard still apply to the link Float32 conversion.

**Loader/verdict adaptation (D-09):** `LoadProgress.stage` union extends from `'fetch'|'parse'|'layout'` to add `'receive'` (bytes flowing) and drop the "Parsing JSON" semantics; `loader.ts` STAGE_LABELS become `Fetching… → Receiving bytes (MB/s)… → Building layout…` with a bytes/edges-decoded counter. `verdict.ts` records the NEW PERF-01 verdict: server fetch+compute ms, stream ms, client decode ms, node+edge counts, peak JS heap — confirming **PASS** (usable on the real dev DB, no swap, 60fps holds after load) vs the recorded 01-03 JSON FAIL.
