# Milestone v1.5: Dgraph Follow-Update Timeout Resilience

**Status:** SHIPPED 2026-06-18
**Phases:** 12
**Total Plans:** 1
**Requirements:** 6/6 delivered (DWRITE-01/02/03/04, OBS-02, TEST-06)

## Overview

v1.5 fixed the production Dgraph follow-update `DeadlineExceeded` failure mode. A slow or oversized `AddFollowers` call no longer aborts the crawler batch when the underlying Dgraph error is transient. Instead, the affected pubkey is treated as a retryable per-pubkey write failure, omitted from `MarkAttempted`, and left stale/eligible for a later retry. Fatal Dgraph write errors still fail loudly.

The milestone preserved DeepFry's data separation and kind-3 correctness constraints: Dgraph still stores ID-only graph data, and `AddFollowers` still performs one all-or-nothing transaction for the full replaceable contact list.

## Success Criteria

1. A simulated or real Dgraph `DeadlineExceeded` from the follow-update path is handled as a transient pubkey/update failure, not a crawler-process abort.
2. Large follow lists are written in bounded units with clear partial-progress behavior and no duplicate/corrupt follow-edge state.
3. Fatal Dgraph write errors still surface loudly and are not retried indefinitely under the transient path.
4. Production logs show enough timing and chunk/pubkey context to diagnose slow `AddFollowers` operations.
5. `make test` passes with regression coverage for timeout classification, partial progress, retry scheduling, and fatal passthrough.

## Milestone Summary

**Key accomplishments:**

- Added `dgraph.IsTransientError`, shared by Dgraph follow writes and main-loop retry behavior. `DeadlineExceeded` and `Unavailable` are transient; `ResourceExhausted` remains fatal.
- Added `FollowUpdateError` and AddFollowers progress tracking with pubkey, follow count, phase, chunk, elapsed time, retry count, and outcome.
- Added bounded child contexts around AddFollowers Dgraph query/mutation/commit units while preserving one final transaction commit.
- Changed `FetchAndUpdateFollows` to return `FetchResult{Hits, SkipAttempt}` so transient write failures continue the batch but remain retry-eligible.
- Added `attemptableBatchKeys` so `MarkAttempted` excludes transient-failed pubkeys.
- Added short tests for classification, progress accounting, transient skip-and-continue, fatal passthrough, and MarkAttempted filtering.

**Key decisions:**

- Transient AddFollowers failures are not retried inside the current batch; they are scheduled naturally by leaving the pubkey stale.
- `AddFollowers` keeps full kind-3 replacement all-or-nothing. No partial follow-edge chunk commits were introduced.
- `ResourceExhausted` remains fatal to preserve the v1.3 anti-livelock decision under indefinite retry semantics.

**Issues resolved:**

- Production failure mode: `failed to add follows: query follower failed: rpc error: code = DeadlineExceeded desc = context deadline exceeded` no longer stops the entire crawler batch when classified transient.
- Integration-tag package build was updated for the current `BackfillNextAttempt(ctx, cadence)` signature so guarded AddFollowers verification can run.

**Issues deferred:**

- Broader crawl-throughput optimization remains deferred to the spike backlog.
- Live-host performance tuning remains outside this milestone.
