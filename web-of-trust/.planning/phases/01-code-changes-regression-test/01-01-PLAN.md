<objective>
Implement Fixes A–D and the §5 regression test from `8pc_crawled.md` — the complete code-side
remedy for the crawler only ever crawling ~8% of known pubkeys.

Purpose: The current `GetStalePubkeys` query sorts by a missing-value predicate (`orderasc:
last_db_update`) and relies on Dgraph's default 1000-row cap, so it returns only already-crawled
accounts and never a single stub. The fix introduces a `last_attempt` predicate (distinct from
`last_db_update`), rewrites selection to pull the never-attempted frontier first via an explicit
`NOT has(last_attempt)` + `first:` query, stamps every queried batch as attempted so un-fetchable
pubkeys age out, and adds a regression test that asserts a stub IS selected.

Output: Updated `pkg/dgraph/dgraph.go` (schema + selection + attempt tracking), updated
`cmd/crawler/main.go` (loop wiring), a new integration test `pkg/dgraph/dgraph_stale_test.go`,
a clean `make build-crawler`, and a passing `make test-integration`.

Scope boundary: Fix E (the one-time DQL backfill, MIG-01) and the §6 live-crawler verification
are Phase 2 / operational steps on the strfry host — OUT of scope here. This plan delivers only
the codebase changes, build, and the automatable integration regression test.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| crawler → Dgraph (gRPC :9080) | Crawler writes graph mutations and reads selection queries. All values written here (UIDs, integer timestamps) are crawler-internal, not event/user payloads. |
| (no new boundary) | This phase adds no network endpoint, no auth surface, no new external input parser. Pubkeys flowing into `MarkAttempted` originate from `GetStalePubkeys` (`has(pubkey)` nodes already in the graph). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering / Injection | `MarkAttempted` n-quad builder (`pkg/dgraph/dgraph.go`) | mitigate | N-quads are built only from Dgraph-resolved UIDs and an integer `ts` — never from raw event/user strings. Pubkeys are resolved to UIDs via `ResolvePubkeysToUIDs` (which itself `strconv.Quote`s inputs) before any interpolation. No untrusted string is concatenated into a query/mutation. |
| T-01-02 | Information Disclosure | logging in crawler loop / dgraph package | accept | New log lines emit only counts and a non-fatal MarkAttempted warning (`%v` on a wrapped error) — no secrets, private keys, or event payloads. Consistent with existing logging conventions (no raw secrets). |
| T-01-03 | Tampering | DQL queries built with `fmt.Sprintf` in `GetStalePubkeys`/`collectStale` | accept | Interpolated values are an `int` limit and an `int64` timestamp only — no string operands — so there is no injection vector. Block names are compile-time constants. |
| T-01-SC | Tampering | go module deps (npm/pip/cargo style installs) | accept | No new external packages are added; the plan reuses already-imported `dgo/v210`, `dgo/v210/protos/api`, and stdlib. No `go get` / package-legitimacy gate required. |

ASVS L1 note: this is a backend graph-write change with no authentication, session, or untrusted
input-parsing surface, so the bulk of ASVS L1 controls are not applicable. The one relevant
control — avoid injection in data-store queries — is satisfied by integer-only interpolation and
UID-only n-quad construction.
</threat_model>
