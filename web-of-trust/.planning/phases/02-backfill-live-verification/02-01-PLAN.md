<objective>
Run the one-time Fix E DQL backfill that seeds `last_attempt` from `last_db_update` on
existing crawled nodes (MIG-01), then verify on the strfry production host that the
frontier-first crawler (shipped in Phase 1) actually converts stub nodes to crawled nodes
in a live run.

Purpose: Without the backfill, the ~15k already-crawled nodes have no `last_attempt`, so
the Phase-1 `NOT has(last_attempt)` frontier query would treat all 189k nodes as frontier
and re-crawl the known set before reaching real stubs. The backfill is the migration that
makes the code fix effective in production. This phase proves the end-to-end fix unblocks
graph growth.

Output: A completed backfill (every `last_db_update` node now has `last_attempt`), a
captured `/tmp/crawler.log` from a 5-minute live run, and recorded baseline/post counts
showing the stub fraction falling.
</objective>

<task type="auto">
  <name>Task 1: Apply schema then run the Fix E one-time backfill (MIG-01)</name>
  <files>(operational — no repo files modified; mutates live Dgraph + ~/deepfry, builds bin/crawler)</files>
  <action>
On the strfry host, execute in this strict order (the ordering is the core MIG-01 dependency — the `last_attempt` predicate MUST exist before the backfill can set it):

1. Build the Phase-1 binary: `cd web-of-trust && make build-crawler`.

2. Apply the schema. `EnsureSchema` runs on crawler startup (`pkg/crawler/crawler.go:76`) and adds `last_attempt: int @index(int)` to the live Dgraph (Fix A, shipped Phase 1). Start the crawler once and let it reach the point past `EnsureSchema`, then stop it — OR confirm the predicate exists via the verify check below. A brief start is sufficient:
   `cd web-of-trust && ./bin/crawler < /dev/null > /tmp/crawler-schema.log 2>&1 & SPID=$!; sleep 20; kill -INT $SPID`

3. Run the Fix E backfill (verbatim from 8pc_crawled.md §4, lines 348-354) — seeds `last_attempt` from `last_db_update` for every existing crawled node:
   ```
   curl -s -H 'Content-Type: application/dql' -X POST localhost:8080/mutate?commitNow=true \
     -H 'Content-Type: application/rdf' --data-binary @- <<'EOF'
   upsert {
     query { nodes as var(func: has(last_db_update)) { ldu as last_db_update } }
     mutation { set { uid(nodes) <last_attempt> val(ldu) . } }
   }
   EOF
   ```

FALLBACK (per §4 lines 356-358): if this Dgraph build rejects the inline HTTP upsert, run the equivalent upsert through Ratel (http://localhost:8000) or via the crawler at startup. Do NOT proceed to Task 2 until the backfill has committed successfully. Implements MIG-01.
  </action>
  <verify>
    <automated>curl -s -H 'Content-Type: application/dql' -X POST localhost:8080/query -d '{ q(func: has(last_db_update)) @filter(NOT has(last_attempt)) { c: count(uid) } }' | grep -q '"c":0'</automated>
  </verify>
  <done>The backfill committed without error AND the verify query returns 0 — i.e. there are zero nodes that have `last_db_update` but lack `last_attempt`. This is Phase 2 Success Criterion 1.</done>
</task>
