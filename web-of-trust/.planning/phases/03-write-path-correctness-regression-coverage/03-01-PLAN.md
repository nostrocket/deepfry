<objective>
Eliminate the chunked-write data-drop bug at its root by moving chunking inside `pkg/dgraph`: `AddFollowers` receives the full follow set in one call, runs the version guard and remove-all-existing-follows exactly once, and batches only the internal followee-resolution query and edge mutations to stay under the ~4MB gRPC limit. Delete `pkg/crawler/chunks.go` and the `>10000` crawler branch so there is a single write path. Add a shared hex-pubkey validator gating every Dgraph pubkey-add site.

Purpose: Chunks 2…N currently re-enter `AddFollowers` carrying the same `createdAt`, trip the `kind3createdAt <= existingKind3CreatedAt` guard at `dgraph.go:165`, and return early — silently dropping all but the first 200 follows for pubkeys with >500 follows. Unifying to one logical write per event makes the bug class impossible while preserving genuine-duplicate dedup (CHUNK-02) and removing the per-iteration `defer cancel()` leak (LEAK-01) by deletion.

Output: `pkg/dgraph/validate.go` (new), rewritten `AddFollowers` + `chunkSlice` helper in `pkg/dgraph/dgraph.go`, deleted `pkg/crawler/chunks.go`, simplified `pkg/crawler/crawler.go`, deduped `cmd/healthcheck/main.go`.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| relay → crawler | kind-3 events (untrusted): pubkey + p-tag followee list arrive from public Nostr relays |
| crawler → pkg/dgraph | `AddFollowers(signerPubkey, createdAt, follows)` crosses into mutation-building code |
| pkg/dgraph → Dgraph | nquads / DQL query strings sent to the gRPC backend |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | `AddFollowers` followee/signer strings + `MarkAttempted` input pubkeys → nquads (`dgraph.go`) | mitigate | `isValidHexPubkey` gate (D-08/D-09) rejects signer / skips followee / skips MarkAttempted input before any pubkey reaches an nquad; `^[0-9a-f]{64}$` makes injection of nquad/DQL metacharacters impossible. `%q`-quoting retained as defense-in-depth. |
| T-03-02 | Denial of Service | large follow-list (>10k) → single transaction (`dgraph.go`) | mitigate | Internal 200-item batching (D-06) keeps each gRPC message under the ~4MB cap; size-scaled timeout (D-07) prevents both premature failure on large lists and unbounded hangs. |
| T-03-03 | Tampering (data integrity) | partial write of a replaceable kind-3 (remove-all then re-add) | mitigate | Single all-or-nothing transaction (D-05): guard + remove-existing + all batched mutations commit once, so a mid-write failure cannot leave a half-emptied follow-list. |
| T-03-04 | Repudiation / silent loss | chunks 2…N discarded by version guard (`dgraph.go:165`) | mitigate | One logical write per event (D-01/D-04) removes the "subsequent chunk same createdAt" case entirely; the guard runs once and only short-circuits genuine duplicates. |
| T-03-05 | Resource exhaustion (leak) | per-iteration `context.WithTimeout` in chunk loop (`chunks.go:39-40`) | mitigate | File deleted (D-02/D-03/LEAK-01); unified path holds at most one live context. |
| T-03-SC | Tampering | npm/pip/cargo installs | accept | No package-manager installs in this phase — all changes use the existing Go module dependency set (no new imports beyond stdlib `regexp`). No supply-chain surface added. |
</threat_model>
