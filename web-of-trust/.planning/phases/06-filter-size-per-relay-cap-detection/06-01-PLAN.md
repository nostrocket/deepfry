  <action>
    1. In the relayState struct, add `filterCap int` as a new field after `failures atomic.Int32` (per D-03 and PATTERNS.md §"Edit 1"). Plain int — not atomic — because only the per-relay goroutine writes it.

    2. In the crawler.Config struct, add `FilterBatchSize int` as a new field. This carries the configured batch size from cmd/crawler into New() and ReconnectRelays() so the notice handler can access the floor value on reconnect.

    3. In the Crawler struct, add `filterBatchSize int` as an unexported field to hold the configured batch size for use in ReconnectRelays() (where cfg is no longer in scope).

    4. Add `handleFilterNotice` as a package-level function (not a method) in crawler.go after the `cleanSubscribeError` function. Signature: `func handleFilterNotice(rs *relayState, notice string, minCap int)`. Implementation: convert notice to lowercase; if it contains both "filter" and "too large", halve rs.filterCap using `max(rs.filterCap/2, minCap)` (Go 1.21+ built-in); log the result. If rs.filterCap is already <= minCap, skip the halving (already at floor). Per D-04/D-05 and PATTERNS.md §"handleFilterNotice helper".

    5. In New(): before the `nostr.RelayConnect` call inside the relay URL loop, create `rs` first (so the closure can capture its pointer): `rs := &relayState{url: url, backoff: initialBackoff, filterCap: cfg.FilterBatchSize}`. Then construct the notice handler as a closure over `rs`: `noticeHandler := nostr.WithNoticeHandler(func(notice string) { handleFilterNotice(rs, notice, 10) })`. Pass `noticeHandler` as the third argument to `nostr.RelayConnect`. After a successful connect assign `rs.conn = relay` and `rs.alive = true` before appending (per D-08 and PATTERNS.md §"Edit 2"). The loop-variable capture issue is avoided because `rs` is created fresh each iteration.

    6. Store `c.filterBatchSize = cfg.FilterBatchSize` when constructing the Crawler value.

    7. In ReconnectRelays(), for each dead relay that has passed its retryAt: construct the same `noticeHandler` closure over the existing `rs` pointer (already a pointer from c.relays), then call `nostr.RelayConnect(ctx, rs.url, noticeHandler)` (per D-08 and PATTERNS.md §"Edit 3"). Use `c.filterBatchSize` as the minCap argument (10 is the hard floor per D-05, but we pass 10 directly in the closure — the filterBatchSize is the configured starting cap, not the floor).

    NOTE on floor constant: the cap floor (10) is a fixed constant in D-05. The `minCap` argument to `handleFilterNotice` is 10, not cfg.FilterBatchSize. cfg.FilterBatchSize is the initial filterCap value assigned at connect time. These are two different values.
  </action>
