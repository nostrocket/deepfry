    1. Extract drainSubscription: create a new method `func (c *Crawler) drainSubscription(ctx context.Context, sub *nostr.Subscription, relayURL string, eventsChan chan<- *nostr.Event) error` containing the existing event-drain `for { select { case event := ... case <-sub.EndOfStoredEvents: ... case <-sub.Context.Done(): ... case <-ctx.Done(): ... } }` loop body from the current queryRelay. The method returns nil on EOSE, ctx.Err() on external cancellation, and &transportError on sub.Context.Done(). Do NOT defer sub.Unsub() inside drainSubscription — the caller manages Unsub().

    2. Update queryRelay signature to accept `rs *relayState` as a new parameter between `relay *nostr.Relay` and `relayURL string` (or after relayURL — choose the position that is least disruptive to existing call sites; `relayURL string` can be dropped in favor of `rs.url` since rs is now available). The simplest approach: replace the `relay *nostr.Relay, relayURL string` parameters with `rs *relayState` only, and use `rs.conn` and `rs.url` inside the function. Check the call site in FetchAndUpdateFollows and update it accordingly.

    3. Implement the chunk loop in queryRelay:
       - `authors := filter.Authors`
       - Loop `for len(authors) > 0`:
         - `cap := rs.filterCap`; if `cap <= 0` set `cap = 10` (safety guard)
         - `chunk := authors; if len(authors) > cap { chunk = authors[:cap] }`
         - `authors = authors[len(chunk):]`
         - `chunkFilter := filter; chunkFilter.Authors = chunk`
         - Record `subscribeStart := time.Now()`
         - Call `relay.Subscribe(ctx, []nostr.Filter{chunkFilter})`
         - On Subscribe error:
           - If `ctx.Err() != nil`: return ctx.Err()
           - Else if `time.Since(subscribeStart) < 500*time.Millisecond` AND `(strings.Contains(err.Error(), "not connected") || strings.Contains(err.Error(), "failed to write"))`: this is a connection-drop-on-REQ (D-09). If `rs.filterCap > 10`: halve `rs.filterCap = max(rs.filterCap/2, 10)`, log "Relay %s: filter rejection drop, halved cap to %d; retrying chunk", re-prepend chunk to authors (`authors = append(chunk, authors...)`), continue. Else (filterCap already == 10): log "Relay %s: filter cap floor reached, marking dead", return `&transportError{err: fmt.Errorf(...)}`.
           - Else: use existing cleanSubscribeError logic — if "not connected"/"failed to write" return transportError; else return subscriptionError.
         - On Subscribe success: call `c.drainSubscription(ctx, sub, rs.url, eventsChan)`. On error from drainSubscription: call `sub.Unsub()`, return the error. On success: call `sub.Unsub()`, continue to next chunk.
       - Return nil after all chunks processed.

    4. Update the FetchAndUpdateFollows call site (currently `c.queryRelay(relayQueryContext, rs.conn, rs.url, filter, eventsChan)`) to match the new signature. If you chose to replace `relay, relayURL` with `rs`, the call becomes `c.queryRelay(relayQueryContext, rs, filter, eventsChan)`.

    Do NOT put fenced code blocks in this action — implementation follows from the behavior description above. Use `max(a, b)` built-in (Go 1.21+).
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| relay→crawler | Relay controls Subscribe error messages and connection timing — both influence filter cap adaptation |
| relay→crawler | A relay can trigger repeated 500ms drops to drive filterCap to floor and cause relay ejection |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-04 | Denial of Service | queryRelay chunk loop — malicious relay causes infinite retry via repeated sub-500ms drops | mitigate | Cap floor=10 limits halving; at floor the relay is marked dead via transportError — loop exits; max iteration count is log2(initialCap/10) ≈ 4 retries then dead |
| T-06-05 | Tampering | Subscribe error message — relay-controlled string influences code path via "not connected"/"failed to write" substring check | accept | These strings come from the go-nostr WebSocket transport layer, not from the relay over the wire; relay cannot inject them via NOTICE or protocol messages |
| T-06-06 | Denial of Service | drainSubscription — relay withholds EOSE indefinitely | accept | Existing 30s timeout context (reduced to 15s in Phase 8) already cancels this path; no change in this plan |
| T-06-SC | Tampering | npm/pip/cargo installs | accept | No new package-manager installs; go.mod unchanged |
</threat_model>
