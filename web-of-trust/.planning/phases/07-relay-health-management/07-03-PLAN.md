<objective>
Close the five interdependent defects 07-VERIFICATION.md found in the Phase 7 relay state machine (all in pkg/crawler/crawler.go, one touching cmd/crawler/main.go). The config layer (07-01) and the parts of 07-02 that passed verification (per-class counters, decay-on-reconnect, probe-up sizing, log consolidation, EjectRelayURL wiring) are VERIFIED and must NOT be redone — this plan only restructures the queryRelay → errorsChan → markRelayDead flow and the New() startup path so the threshold mechanism actually governs ejection.

The fix is one coherent restructure with a single root cause: markRelayDead is currently called from inside queryRelay (which runs in a per-relay goroutine). That single fact produces CR-01 (cascade: markRelayDead closes the conn, the loop continues on the dead conn and re-triggers rejection), CR-02 (data race: goroutines mutate c.relays without a mutex), and is entangled with WR-01 (floor-reached returns transportError, mis-classified) and WR-03 (defer inside the loop). The verifier's PREFERRED remediation removes markRelayDead from the goroutine entirely: queryRelay returns typed errors, and the single-threaded dispatcher in FetchAndUpdateFollows is the sole caller of markRelayDead. CR-03 (startup eject bypasses thresholds) is a separate localized fix in New().

Purpose: With these fixes, a single real filter rejection bumps failFilterRej by 1 (not 0→3), the threshold of 3 governs ejection over multiple cycles, there is no data race, and a transient startup outage cannot brick the crawler by ejecting all relays. This is the difference between "relays ejected on first failure" and the phase goal "relays that repeatedly fail are automatically removed."

Output: a filterRejectionError type; a queryRelay that returns (never ejects); a dispatcher that classifies subscriptionError→classSubFlap / transportError→classTransport / filterRejectionError→classFilterRej; a New() that keeps failed-startup relays in the pool with alive=false; a hoisted probe-flag defer; and tests that drive the real seam (not inline copies) so these exact regressions are caught by go test -race.

This is surgery on one cohesive flow; the two tasks are ordered by data flow (restructure the query/dispatch/startup paths → tests that drive the real seam) and must not be reordered. Task 2's tests are co-located with Task 1's edits because they exercise the same code.
