<objective>
Make Phase 12 deliver Dgraph follow-update resilience end to end: a transient Dgraph `DeadlineExceeded` during `AddFollowers` must not abort the crawler batch, must not stamp the affected pubkey as a clean hit or miss, and must leave the existing graph state canonical for later retry (DWRITE-01, DWRITE-03). Keep fatal write errors loud (DWRITE-04). Bound large-list write work into per-window operations with observable progress while preserving the current all-or-nothing transaction for kind-3 replaceable semantics (DWRITE-02). Add production diagnostics and regression tests (OBS-02, TEST-06).

Purpose: The 2026-06-18 production failure occurred after relay fetch succeeded; Dgraph follow-update/bookkeeping dominated the batch and a single `DeadlineExceeded` stopped progress. This plan changes the write failure from process-level fatal to pubkey-level retryable only when the Dgraph code is transient.

Output: Modified Dgraph, crawler, and main-loop code plus unit tests that prove transient skip, retry eligibility, fatal passthrough, and progress accounting.
</objective>
