<objective>
Eliminate the dominant per-batch read-path cost in GetStalePubkeys (≈39s of an ~80s batch),
which today recomputes count(~follows) over the entire ~1.3M never-attempted frontier on every
call purely to order it. Introduce a stored, indexed follower_count predicate on Profile,
maintain it cheaply as follow edges change in AddFollowers, supply a one-time operator backfill
for existing nodes, and re-point the frontier/aged ordering to read the stored value.

Purpose: frontier ordering reads a value instead of recomputing an aggregate, removing the
single largest cost in the crawl loop (DSCALE-01) while keeping ordering accurate as edges
change (DSCALE-03) and providing live before/after evidence (TEST-03).

Output: modified pkg/dgraph/dgraph.go (schema + read path + write maintenance + backfill method
+ pure delta helper), a new cmd/backfill-follower-count CLI, a Makefile build target, and a new
test file with unit + integration coverage.

Accuracy contract (governs every task): follower_count is an ORDERING HINT, not authoritative
(per D-context Area 1). Small drift is tolerated. Do NOT add exactness machinery, reconciliation
passes, or authoritative recomputation on the hot path — cheap delta maintenance plus the
operator backfill is the intended design.
</objective>
