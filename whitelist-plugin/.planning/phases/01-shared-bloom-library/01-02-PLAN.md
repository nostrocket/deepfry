<objective>
Add the benchmark gate for `pkg/bloom`: size-swept `Filter.Contains` hit/miss benchmarks and a `Builder.Build` benchmark, plus an assertion that `Filter.Contains` is `0 allocs/op`. This is the second of the three mandatory de-risking checks from CONTEXT <specifics> — it validates the D-08 claim that passing `k[:]` from a `[32]byte` is alloc-free and keeps the ~10k-events/sec plugin hot path at zero allocations.

Purpose: The plugin hot path (Phase 3) calls `Filter.Contains` per event. If `k[:]` ever escaped to the heap, throughput and GC pressure would regress. This benchmark is the regression guard that proves the hot path is alloc-free before Phases 2/3 depend on it.

Output: `pkg/bloom/bloom_bench_test.go`. The 0-allocs guard test passes and benchmarks run clean.
</objective>
