<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| bloom-gate plugin (Phase 3) → server `GET /bloom` | untrusted client polls; reads serialized filter + ETag |
| operator YAML (`~/deepfry/whitelist.yaml`) → server config | operator-controlled `bloom_fp_rate` value |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | handleBloom If-None-Match header | accept | header only selects 200 vs 304; never used to index memory or build a response body — no injection surface |
| T-02-02 | Denial of Service | handleBloom per-request work | mitigate | bytes pre-serialized once per generation (D-05); handler does a single atomic Load + one Write, no per-request build; build runs off-request on the refresher goroutine |
| T-02-03 | Information Disclosure | served filter bytes | accept | filter encodes only whitelist membership (already public via /check); ~1e-6 FP leak is a deliberate design tolerance |
| T-02-04 | Denial of Service | bloom_fp_rate config (extreme small value → huge filter) | accept | value is operator-set in a trusted local YAML, not remote input; whitelist ~1.5M keys, default 1e-6 sizing is bounded |
| T-02-SC | Tampering | npm/pip/cargo installs | mitigate | no new external packages added this phase (pkg/bloom + bits-and-blooms already vetted in Phase 1); no install step |
</threat_model>
