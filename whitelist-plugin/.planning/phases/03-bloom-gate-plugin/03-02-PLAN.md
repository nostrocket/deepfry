  <behavior>
    - 200 path: given an httptest server returning 200 with a valid DFBF body and an ETag, fetch() calls bloom.ReadFilter on the body, stores the filter into the checker (checker now ready, membership matches the served filter), AND writes the bytes to bloom_path such that a subsequent bloom.ReadFilter of the file succeeds.
    - 304 path: given a server returning 304, fetch() does not call Store and does not create/modify bloom_path; the previously held filter remains the queried one.
    - If-None-Match: when a filter is already held, the request carries If-None-Match equal to the held filter's ETag(); when none is held, the header is absent.
    - 503 path: a server returning 503 {"status":"loading"} is treated as transient — no swap, no disk write, no fatal error; the last good filter (if any) stays.
    - transport error path: an unreachable server leaves the last good filter in place; with no last-good filter the checker remains un-ready (plugin holds).
    - corrupt-200 path: a 200 body that fails bloom.ReadFilter (e.g. truncated) is discarded — no Store, and bloom_path is left as its prior valid contents (or absent); the .tmp file does not survive as bloom_path.
    - disk-first cold start: with a valid bloom.dfbf already on disk, Start() loads it and the checker is ready before any HTTP call (test can point serverURL at a closed port and still see readiness from disk).
    - atomic write: after a successful persist, bloom_path exists and no `${bloom_path}.tmp` remains.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| server /bloom -> fetcher | The HTTP response (octet-stream body + status + ETag) is semi-trusted input crossing into the plugin. A truncated/corrupt/oversized body must not corrupt the persisted file or crash the plugin. |
| fetcher -> bloom_path (disk) | The fetcher writes under ~/deepfry/. A crash mid-write must never leave a torn bloom.dfbf. |
| StrFry stdin events -> event loop | Untrusted event JSONL; parsed by the reused JSONLIOAdapter exactly as the whitelist plugin (unchanged surface). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-04 | Tampering | corrupt/truncated /bloom body | mitigate | Parse-before-persist: bloom.ReadFilter (magic + formatVersion + payloadLen bound-check) validates before any swap or disk write (D-07); failures discard and keep prior state. |
| T-03-05 | Tampering | torn bloom.dfbf on crash | mitigate | Atomic temp+rename: write `${bloom_path}.tmp` then os.Rename (atomic same-fs); a crash leaves stale-valid or orphan .tmp, never a torn file (D-08). |
| T-03-06 | Denial of Service | oversized payloadLen in /bloom body | mitigate | bloom.ReadFilter caps declared payloadLen at maxPayloadBytes (1 GiB) before allocating (Phase-1 D-07), so a malicious body cannot OOM the plugin. |
| T-03-07 | Denial of Service | server down / 503 storm | mitigate | Keep-last-good + bounded retries then wait for next interval (D-10/D-11); never hammers a down server, never degrades decisions on staleness. |
| T-03-08 | Spoofing | events decided with no filter | mitigate | Cold-start hold (GATE-06/D-06): checker withholds decisions until a filter exists from disk or server — fail-closed, no event admitted un-vetted. |
| T-03-SC | Tampering | npm/pip/cargo installs | accept | No new package-manager installs: fetcher uses stdlib net/http + os + existing in-module pkg/bloom; cmd/bloom uses stdlib + existing pkg/handler, pkg/config, pkg/bloomgate. No legitimacy gate required. |
</threat_model>
